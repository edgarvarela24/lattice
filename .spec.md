> **Version:** 1.2.0-draft  
> **Status:** Pre-implementation design spec  
> **Date:** March 2026

## 1. What Lattice Is

Lattice is a **compiled TypeScript framework** built on fine-grained signals, deep reactive proxies, and compiled tagged templates. It combines Svelte's "signals as values" DX with Solid's fine-grained reactivity model — in plain `.ts` files, with no special file formats, no virtual DOM, and no `.value`.

The signal engine is platform-agnostic. Components render to DOM in the browser. The same reactive primitives work on the server, enabling a unified fullstack model for SSR, hydration, and resumability.

```typescript
import { signal, html } from 'lattice';

export function Counter() {
  let count = signal(0);

  return html` <button @click=${() => count++}>${count}</button> `;
}
```

You write `count++`. The compiler transforms it into a signal write. You write `${count}` in a template. The compiler generates a direct signal-to-DOM subscription at build time. One value changes, one DOM node updates. Nothing else runs.

### 1.1 Why Lattice Exists

**React** is simple to start but painful at scale: stale closures, `useEffect` footguns, a virtual DOM that diffs the entire tree when one value changes, and manual memoization that the React Compiler is only now beginning to automate.

**Svelte** proved that compiled signals with value semantics are the best DX in frontend. But reactivity only works in `.svelte` files, shared state requires `.svelte.ts` files, and the component format is a custom language — not plain TypeScript.

**Solid** proved that fine-grained signals with no virtual DOM deliver the best performance. But signals are containers with function-call getters (`count()` not `count`), destructuring breaks reactivity, and there's no compiler transform for signal DX.

**Lattice** takes the best of both: Svelte's "signals are values" DX powered by a compiler, Solid's fine-grained reactivity with no virtual DOM, deep reactive proxies so mutation works everywhere — all in plain TypeScript files, with a clear path to fullstack.

### 1.2 Core Principles

| Principle                       | Meaning                                                                                                     |
| ------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| **Signals are values**          | `count` is a number. You increment it with `++`. You pass it to functions. The compiler handles the rest.   |
| **Mutation works**              | `count++`, `todos.push(item)`, `user.name = 'x'` — all reactive, all tracked.                               |
| **One way to do things**        | `signal()` everywhere. Same syntax in components, shared modules, server code, anywhere. No special cases.  |
| **Plain TypeScript**            | No `.svelte`, no `.jsx`, no custom file formats. Components are functions in `.ts` files.                   |
| **Five concepts**               | `signal`, `computed`, `effect`, `watch`, `html`. That's the framework.                                      |
| **Compiler-visible reactivity** | The compiler knows every dependency. It shows you, warns you, and catches mistakes.                         |
| **Platform-agnostic core**      | The signal engine runs anywhere JavaScript runs. The template layer targets the DOM. The boundary is clean. |

### 1.3 The Public API

```typescript
import { signal, computed, effect, watch, html } from 'lattice';
```

| Primitive                 | Purpose                                                                                                     |
| ------------------------- | ----------------------------------------------------------------------------------------------------------- |
| `signal(value)`           | Reactive state. Compiler transforms reads/writes so it behaves as a plain value.                            |
| `computed(fn)`            | Computed derivation. Lazy, cached, equality-checked. Re-evaluates when dependencies change.                 |
| `effect(fn)`              | Reactive side effect. Auto-tracks dependencies. Compiler shows you what it depends on.                      |
| `watch(source, callback)` | Explicit side effect. Declared source, no auto-tracking in the callback. For when you need precise control. |
| `` html`...` ``           | Compiled tagged template. Generates direct signal-to-DOM subscriptions at build time.                       |

Five words. Everything else is plain TypeScript.

### 1.4 Why These Names

The names are deliberately familiar. `signal` is the industry term (Solid, Angular, Preact). `computed` is what Vue, Solid, Angular, and MobX call it. `effect` is what Solid, Preact, and Angular call it. `watch` is what Vue calls it. Lattice's differentiator is the DX and the compiler, not the vocabulary. A developer coming from any modern framework already knows what these do.

## 2. Signals as Values

The core DX innovation: signals behave as plain values, not containers.

### 2.1 The Compiler Transform

The developer writes:

```typescript
let count = signal(0);
count++;
console.log(count);
```

The compiler outputs:

```typescript
import { signal } from '@lattice/signals';
const __count = signal(0);
__count.value++;
console.log(__count.value);
```

Every read of `count` becomes a `.value` read. Every write becomes a `.value` write. The developer never sees `.value`. The variable behaves as its contained type — `count` is a `number`, not a `Signal<number>`.

### 2.2 Where Signals Work

Signals work identically everywhere. No special files. No special scopes.

**In components:**

```typescript
import { signal, html } from 'lattice';

export function Counter() {
  let count = signal(0);
  return html` <button @click=${() => count++}>${count}</button> `;
}
```

**In shared modules:**

```typescript
// stores/counter.ts
import { signal } from 'lattice';

export let count = signal(0);
export function increment() {
  count++;
}
export function reset() {
  count = 0;
}
```

**In utility/logic files:**

```typescript
// composables/useDebounce.ts
import { signal, effect } from 'lattice';

export function useDebounce<T>(source: () => T, ms: number) {
  let debounced = signal(source());

  effect(() => {
    const val = source();
    const timeout = setTimeout(() => (debounced = val), ms);
    return () => clearTimeout(timeout);
  });

  return debounced;
}
```

**On the server:**

```typescript
// server/monitor.ts
import { signal, computed, effect } from 'lattice';

export let activeConnections = signal(0);
export let serverHealthy = computed(() => activeConnections < 1000);

effect(() => {
  if (!serverHealthy) {
    logger.warn(`High connection count: ${activeConnections}`);
  }
});
```

**Same pattern. Same syntax. Everywhere.**

### 2.3 What the Compiler Tracks

The compiler transforms reads and writes for any variable initialized with `signal()` or `computed()`. It does this by tracking the variable binding through its lexical scope:

- `let x = signal(0)` — `x` is a signal binding. All reads/writes transformed.
- `let y = computed(() => x * 2)` — `y` is a computed binding. Reads transformed (writes are an error).
- `let z = 5` — `z` is a plain variable. Not transformed. Not reactive.

The rule is explicit: **only variables initialized with `signal()` or `computed()` are transformed.** Everything else is plain TypeScript.

### 2.4 Compiler Edge Cases & Rules

The compiler's transform is based on one principle: **signal bindings are reactive through the variable name.** When a signal value leaves the variable binding, it becomes a snapshot.

#### Object Shorthand — Snapshot Semantics

```typescript
let count = signal(0);
const obj = { count }; // { count: 0 } — captures value, not binding

// Compiled:
const obj = { count: __count.value };
```

This is intentional. If `obj.count` were a live binding, objects passed to third-party libraries (`JSON.stringify(obj)`, `Object.keys(obj)`, etc.) would behave unexpectedly. The rule is simple: put a signal in an object, get a snapshot.

**If you want reactive objects, use `signal()` directly:**

```typescript
let state = signal({ count: 0 });
state.count++; // reactive — deep proxy handles this
```

#### Destructuring — Not Transformed

```typescript
let [a, b] = [signal(0), signal(1)]; // a and b are NOT signal bindings
```

Only variables directly initialized with `signal()` or `computed()` are transformed. Destructuring breaks the pattern. Write two declarations instead:

```typescript
let a = signal(0);
let b = signal(1);
```

#### Reassignment

```typescript
let count = signal(0);
count = 5; // → __count.value = 5 (signal write)
count = signal(10); // COMPILER ERROR: Cannot reassign signal binding to new signal
```

#### Factory Returns — Explicit Accessors (v1)

```typescript
// Returning signal bindings from a function captures values, not bindings:
function createCounter() {
  let count = signal(0);
  return { count }; // { count: 0 } — snapshot, not reactive
}

// To expose reactive state, use getter/setter:
function createCounter() {
  let count = signal(0);
  return {
    get count() {
      return count;
    },
    set count(v) {
      count = v;
    },
    increment() {
      count++;
    },
  };
}
```

This is verbose but explicit. The compiler may add automatic getter/setter generation for return objects containing signal bindings in a future version, but v1 uses explicit accessors to keep the compiler simple and predictable.

#### Closures in Async — Works Correctly

```typescript
let count = signal(0)
setTimeout(() => count++, 1000)     // ✅ works — closure captures __count
await fetch(...).then(() => count++) // ✅ works — same reason
```

The closure captures the `__count` signal object (a `const`), and `count++` inside the closure is transformed to `__count.value++`. Since `__count` never changes (only `.value` does), closures work correctly in all async contexts.

#### Exports — Both Files Must Be Compiled

```typescript
// stores/counter.ts (compiled)
export let count = signal(0);

// Compiled output exports the signal object:
export { __count as count };

// components/Counter.ts (compiled)
import { count } from '../stores/counter';
// Compiler knows 'count' is a signal import → transforms reads/writes
```

The importing file's compiler recognizes signal exports and transforms reads/writes accordingly. Non-Lattice code importing a signal sees the raw signal object with `.value` — this is the interop escape hatch.

### 2.5 Passing Signals

When a signal variable is passed to a function, the compiler passes the current value:

```typescript
let count = signal(0);
console.log(count); // passes the value (number)
someFunction(count); // passes the value (number)
```

If you need to pass the signal itself (for composition), use an explicit accessor pattern:

```typescript
// Pass a getter — the function can read the signal reactively
useDebounce(() => count, 300);
```

This is the same pattern Vue, Solid, and Angular use for reactive composition. The accessor `() => count` is transformed by the compiler into `() => __count.value`, which creates a live binding.

## 3. Computed State

`computed()` creates a computed value that automatically updates when its dependencies change. It is lazy (only evaluates when read), cached (returns the cached value if dependencies haven't changed), and equality-checked (doesn't propagate if the new value equals the old value).

```typescript
import { signal, computed, html } from 'lattice';

export function TodoList() {
  let todos = signal<Todo[]>([]);
  let filter = signal<'all' | 'active' | 'done'>('all');

  let visible = computed(() =>
    filter === 'all' ? todos : todos.filter((t) => (filter === 'done' ? t.done : !t.done)),
  );

  let remaining = computed(() => todos.filter((t) => !t.done).length);

  return html`
    <p>${remaining} items left</p>
    <ul>
      ${visible.map((todo) => html` <li key=${todo.id}>${todo.text}</li> `)}
    </ul>
  `;
}
```

### 3.1 Computed Rules

- The function passed to `computed()` must be a pure computation. No side effects.
- `computed` bindings are read-only. Writing to a `computed` variable is a compiler error.
- `computed` uses dynamic dependency tracking internally (via the signal engine's observer system) to discover which signals the function reads.
- Custom equality can be provided: `computed(fn, { equals: (a, b) => a.id === b.id })`.

### 3.2 Error Handling in Computed

If a computed's function throws, the error is cached:

- First evaluation throws → error is cached. Subsequent `.value` reads re-throw the cached error.
- When a dependency changes, the computed re-evaluates. If the new evaluation succeeds, the error clears. If it throws again, the new error is cached.
- This avoids infinite re-evaluation of broken functions and naturally clears when inputs change.

### 3.3 Circular Dependency Detection

If computed A reads computed B which reads computed A, the runtime detects the re-entrant read and throws a clear error:

```
Error: Circular dependency detected in computed
```

Self-referencing computeds (a computed that reads itself) are also detected. After the error, the system is in a consistent state — no flags are corrupted.

## 4. Reactive Side Effects

Lattice provides two primitives for side effects: `effect` for auto-tracked reactivity, and `watch` for explicit source declaration. Both are tied to the ownership tree and automatically cleaned up when their scope disposes.

### 4.1 Effect

`effect()` is the primary side effect primitive. It executes immediately, auto-tracks which signals are read inside it, and re-executes when any of those signals change.

```typescript
import { signal, effect } from 'lattice';

let count = signal(0);

// Runs immediately, re-runs when count changes
effect(() => {
  document.title = `${count} items`;
});
```

**Cleanup:** Returning a function from `effect` registers a cleanup that executes before the next re-run or on disposal.

```typescript
effect(() => {
  const controller = new AbortController();

  fetch(`/api/search?q=${query}`, { signal: controller.signal })
    .then((res) => res.json())
    .then((data) => (results = data));

  return () => controller.abort();
});
```

**Common patterns:**

```typescript
// DOM sync
effect(() => {
  document.title = `${count} items`;
});

// Data fetching with cancellation
effect(() => {
  if (query.length < 3) {
    results = [];
    return;
  }

  const controller = new AbortController();
  loading = true;

  fetch(`/api/search?q=${query}`, { signal: controller.signal })
    .then((res) => res.json())
    .then((data) => {
      results = data;
      loading = false;
    })
    .catch((err) => {
      if (err.name !== 'AbortError') {
        error = err;
        loading = false;
      }
    });

  return () => controller.abort();
});

// Debouncing
effect(() => {
  const timeout = setTimeout(() => (debouncedQuery = query), 300);
  return () => clearTimeout(timeout);
});

// LocalStorage sync
effect(() => {
  localStorage.setItem('theme', theme);
});

// Polling
effect(() => {
  fetchPrice(symbol).then((p) => (price = p));
  const interval = setInterval(() => {
    fetchPrice(symbol).then((p) => (price = p));
  }, 5000);
  return () => clearInterval(interval);
});

// WebSocket
effect(() => {
  const ws = new WebSocket(`wss://api.example.com/chat/${room}`);
  ws.onopen = () => (connected = true);
  ws.onclose = () => (connected = false);
  ws.onmessage = (e) => messages.push(JSON.parse(e.data));
  return () => ws.close();
});
```

### 4.2 Compiler-Enhanced Dependency Visibility

The compiler knows which variables are signal bindings. When it sees signal reads inside an `effect()`, it can:

**Show dependencies in IDE/build output:**

```
// Compiler annotation:
// Dependencies: [query, isSearching]
effect(() => {
  if (isSearching) {
    fetchResults(query)
  }
})
```

**Warn about accidental dependencies:**

```
Warning: effect() reads signal 'userName' which appears to be a debug statement.
  If intentional, ignore. If not, use peek() to read without tracking.
  → components/Search.ts:14  console.log(userName)
```

**Detect dependency changes between edits:**

```
Info: effect() dependency set changed. Previously: [query]. Now: [query, isSearching].
  → components/Search.ts:12
```

This transforms the traditional footgun of auto-tracked effects into a transparent, debuggable system. The compiler sees what the runtime tracks.

### 4.3 Watch

`watch()` is the explicit-source alternative. Use it when you need precise control over what triggers re-execution, or when you need access to old and new values.

```typescript
import { signal, watch } from 'lattice';

let query = signal('');

// Explicit source — only re-runs when query changes
watch(
  () => query,
  (newQuery, oldQuery) => {
    analytics.track('search_changed', { from: oldQuery, to: newQuery });
  },
);

// Multiple explicit sources
watch(
  () => [firstName, lastName],
  ([first, last]) => {
    analytics.track('name_changed', { first, last });
  },
);

// Cleanup
watch(
  () => query,
  (q) => {
    const timeout = setTimeout(() => search(q), 300);
    return () => clearTimeout(timeout);
  },
);
```

### 4.4 Effect vs Watch

|                         | `effect`                                                           | `watch`                                                                                                                              |
| ----------------------- | ------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------ |
| **Dependency tracking** | Automatic — reads inside the function are tracked                  | Explicit — you declare the source                                                                                                    |
| **Runs on init**        | Yes                                                                | No — runs on first change                                                                                                            |
| **Old/new values**      | No                                                                 | Yes — callback receives `(newValue, oldValue)`                                                                                       |
| **When to use**         | Default choice. DOM sync, data fetching, any reactive side effect. | When you need to react to a _specific_ signal change, need old/new comparison, or want to ignore other signals read in the callback. |

**Rule of thumb:** Start with `effect`. Use `watch` when you need its specific capabilities.

### 4.5 Effect Rules

- `effect` executes immediately on creation, then re-executes when tracked dependencies change.
- The function body is auto-tracked: any signal read inside it becomes a dependency.
- Returning a function registers cleanup that runs before re-execution or on disposal.
- `effect` is tied to the ownership tree. Component unmount → automatic cleanup.
- Signals written inside `effect` do not cause the same `effect` to re-trigger (no infinite loops).

### 4.6 Watch Rules

- The first argument is a **source function** — a getter that returns the value(s) to watch. The compiler transforms signal reads inside it.
- The second argument is a **callback** that runs when the source value changes. It receives the new and old values. The callback is NOT auto-tracked — signal reads in the callback do not become dependencies.
- `watch` does not run on initialization. It runs on the first change.
- Returning a function from the callback registers cleanup.
- `watch` is tied to the ownership tree.

## 5. Deep Reactivity

Lattice uses deep reactive proxies. Mutation works at any depth.

### 5.1 What Is Reactive

All of the following trigger updates:

```typescript
// Top-level assignment (compiler-transformed signal)
count = 5;
count++;

// Nested property assignment (deep proxy)
state.user.name = 'Alice';
state.settings.notifications.email = false;

// Array methods (deep proxy)
state.items.push(newItem);
state.items.splice(2, 1);
state.items[3] = updatedItem;
state.items.sort((a, b) => a.name.localeCompare(b.name));

// Deletion
delete state.user.middleName;
```

### 5.2 How It Works

When a `signal()` is initialized with an object or array, the value is wrapped in a recursive Proxy. Each property access path maps to a signal internally. Reading a path subscribes the reader. Writing a path notifies all subscribers of that path and any parent paths.

```typescript
let user = signal({ name: 'Alice', address: { city: 'NYC' } });

user.name = 'Bob'; // notifies subscribers of user.name and user
user.address.city = 'LA'; // notifies subscribers of user.address.city, user.address, and user
```

### 5.3 Collection-Level Tracking Strategy

The naive approach to deep proxying — one Proxy per nested object — creates performance problems at scale. An array of 10,000 objects would create 10,000 individual Proxy instances, each with their own signal maps and dependency sets. Iterating with `.filter()` or `.map()` would trigger 10,000+ proxy trap invocations and register 10,000+ individual dependencies.

Lattice uses a unified strategy based on one principle: **the proxy lives on the container, not on the elements.**

#### How It Works

Instead of wrapping every nested object in its own Proxy with its own signal infrastructure, the top-level `signal()` creates a single **reactive store** that tracks all access paths. When a nested object or array element is accessed, the store returns a lightweight **view** — a Proxy that routes reads and writes back through the store's centralized tracking.

Views are stateless. They don't have their own signal maps or dependency sets. Creating a view is cheap — one Proxy creation with a shared handler, no signal allocation. The store holds the raw (unproxied) data and a map of path-based signals.

#### Two Granularities, Chosen Automatically

The store automatically selects tracking granularity based on access pattern:

| Access Pattern                 | Tracking Level            | Example                                          |
| ------------------------------ | ------------------------- | ------------------------------------------------ |
| **Individual property access** | Fine-grained (path-based) | `todos[5].done` → tracks `todos.5.done`          |
| **Iteration**                  | Coarse (collection-level) | `todos.filter(...)` → tracks `todos` collection  |
| **Nested object property**     | Fine-grained (path-based) | `user.address.city` → tracks `user.address.city` |

When `.filter()`, `.map()`, `.forEach()`, or other iteration methods are called on a reactive array, the store registers a single "collection iterated" dependency rather than N individual element dependencies. The iteration callback receives elements through lightweight views that route writes back to the store, but the dependency is on the collection as a whole.

This means:

- `todos.filter(t => !t.done)` registers **1 dependency** (collection-level), not 10,000
- `todos[5].done = true` triggers the collection dependency, causing the filter to re-run
- This is not "over-notifying" — the filter reads all elements anyway, so it must re-run regardless of which element changed

Individual element access (`todos[5].done`) still gets fine-grained tracking. A `computed` that only reads one specific element won't re-run when a different element changes.

#### Why This Works for All Cases

```typescript
// Primitives array — proxy on the array, raw values returned
let numbers = signal([1, 2, 3, 4, 5]);
numbers[2] = 10; // fine-grained: index 2 changed
numbers.map((n) => n * 2); // coarse: collection iterated

// Array of objects — proxy on the array, views for elements
let todos = signal<Todo[]>([]);
todos[0].done = true; // fine-grained: path todos.0.done
todos.filter((t) => !t.done); // coarse: collection iterated

// Nested object — proxy on the root, views for nested objects
let user = signal({ name: 'Alice', address: { city: 'NYC' } });
user.name = 'Bob'; // fine-grained: path user.name
user.address.city = 'LA'; // fine-grained: path user.address.city
```

One strategy. The developer never thinks about which tracking mode is active.

#### Shallow Escape Hatch

For performance-critical cases where even collection-level tracking overhead is too much:

```typescript
let items = signal<Item[]>([], { shallow: true });
// Array itself is reactive (push/splice/assignment trigger updates)
// Individual item properties are NOT reactive through the proxy
// items[0].done = true → NOT tracked (use with caution)
```

This should be designed into the API from day one even if not implemented immediately, to avoid breaking changes later.

### 5.4 Identity Semantics

Views maintain stable identity:

```typescript
const a = user.address; // returns view
const b = user.address; // returns same view
a === b; // true
```

The same object is never double-proxied. Non-extensible and frozen objects are not proxied (they can't be reactive — the compiler warns if detected).

### 5.5 Batching

Multiple synchronous mutations batch into one propagation:

```typescript
import { batch } from 'lattice';

batch(() => {
  theme = 'dark';
  sidebarOpen = false;
  activeTab = 'logs';
});
// → ONE propagation, not three
```

Batching also occurs automatically within event handlers and `effect`/`watch` callbacks.

### 5.6 Glitch-Free

The signal graph ensures computed values never see inconsistent intermediate states. If signal `s` feeds both `computed A` and `computed B`, and an `effect` reads both, it sees consistent values from a single propagation — never a mix of old and new.

The current implementation achieves this through eager evaluation in computed's `notify` — when a dependency changes, an observed computed immediately re-evaluates so downstream effects see fresh values. A future optimization could use topological sorting (processing nodes in dependency-depth order) to avoid unnecessary intermediate evaluations in deep computed chains, but the current approach is correct and simpler.

## 6. Components

Components are functions that return an `html` tagged template. No classes. No decorators. No special file format.

### 6.1 Basic Component

```typescript
import { signal, html } from 'lattice';

export function Counter() {
  let count = signal(0);

  return html` <button @click=${() => count++}>${count}</button> `;
}
```

### 6.2 Props

Components receive props as function arguments:

```typescript
import { signal, html } from 'lattice';

interface CounterProps {
  initial?: number;
  label?: string;
}

export function Counter({ initial = 0, label = 'Count' }: CounterProps = {}) {
  let count = signal(initial);

  return html` <button @click=${() => count++}>${label}: ${count}</button> `;
}
```

Using a component:

```typescript
import { html } from 'lattice';
import { Counter } from './Counter';

export function App() {
  return html`
    <div>
      <${Counter} initial=${5} label=${'Clicks'} />
      <${Counter} />
    </div>
  `;
}
```

### 6.3 Children / Slots

Components can accept children:

```typescript
import { html } from 'lattice';

export function Card({ title }: { title: string }) {
  return html`
    <div class="card">
      <h2>${title}</h2>
      <div class="card-body">
        <slot />
      </div>
    </div>
  `;
}

// Usage
html`
  <${Card} title=${'Hello'}>
    <p>This goes in the slot.</p>
  </${Card}>
`;
```

### 6.4 Lifecycle

Components have two lifecycle hooks, available as imports:

```typescript
import { signal, html, onMount, onDestroy } from 'lattice';

export function Timer() {
  let elapsed = signal(0);
  let interval: number;

  onMount(() => {
    interval = setInterval(() => elapsed++, 1000);
  });

  onDestroy(() => {
    clearInterval(interval);
  });

  return html`<p>Elapsed: ${elapsed}s</p>`;
}
```

- `onMount(fn)` — runs after the component's DOM is created and inserted. The function can return a cleanup function (equivalent to also registering `onDestroy`).
- `onDestroy(fn)` — runs when the component is removed from the DOM. All signal subscriptions and `effect`/`watch` calls are automatically cleaned up by the ownership system — `onDestroy` is only needed for non-Lattice resources (timers, event listeners, websockets, etc).

### 6.5 Scoped Styles

The `css` tagged template provides component-scoped styles:

```typescript
import { signal, html, css } from 'lattice';

const styles = css`
  button {
    background: var(--color-primary);
    color: white;
    padding: 8px 16px;
    border-radius: 4px;
  }
  button:hover {
    background: var(--color-primary-dark);
  }
`;

export function Counter() {
  let count = signal(0);

  return html` <button @click=${() => count++}>${count}</button> `;
}
```

The compiler:

1. Extracts CSS at build time (no runtime injection).
2. Scopes selectors with a unique component attribute (e.g., `button[data-l-counter]`).
3. Adds the scoping attribute to the component's root elements.
4. Emits static CSS files that the browser caches normally.

## 7. Templates

Templates use `html` tagged template literals — standard JavaScript. The compiler parses them at build time and emits optimized DOM operations.

### 7.1 What the Compiler Generates

```typescript
// You write:
html`
  <div class=${theme}>
    <span>${count}</span>
    <p>${name}</p>
  </div>
`;

// Compiler generates (conceptually):
function template() {
  const div = createElement('div');
  const span = createElement('span');
  const p = createElement('p');
  div.append(span, p);

  // Direct subscriptions — no effect wrappers
  __theme.subscribe((val) => (div.className = val));
  __count.subscribe((val) => (span.textContent = val));
  __name.subscribe((val) => (p.textContent = val));

  return div;
}
```

Each DOM node subscribes to exactly the signals it reads. When `count` changes, only the span updates. The div and p are untouched. No virtual DOM. No diffing.

For template expressions that involve multiple signals, the compiler generates a `computed` and subscribes to that:

```typescript
// You write:
html`<span>${firstName} ${lastName}</span>`;

// Compiler generates:
const __expr = computed(() => `${__firstName.value} ${__lastName.value}`);
__expr.subscribe((val) => (span.textContent = val));
```

### 7.2 Template Bindings

```typescript
// Text interpolation
html`<span>${count}</span>`;

// Attribute binding
html`<div class=${theme}></div>`;

// Boolean attribute
html`<button ?disabled=${isLoading}>Submit</button>`;

// Property binding
html`<input .value=${query} />`;

// Event handler
html`<button @click=${() => count++}>Click</button>`;

// Component rendering
html`<${Counter} initial=${5} />`;
```

### 7.3 Template Purity

Template expressions must be pure reads. Mutations live in event handlers.

```typescript
// ✅ Pure reads — allowed in template expressions
html`<span>${count}</span>`;
html`<span>${count * 2}</span>`;
html`<span>${formatDate(createdAt)}</span>`;
html`<span>${items.length} items</span>`;

// ❌ Side effects — compiler errors
html`<span>${count++}</span>`;
html`<span>${items.push(x)}</span>`;

// ✅ Mutations in event handlers — always allowed
html`<button @click=${() => count++}>Click</button>`;
html`<button @click=${() => items.push(newItem)}>Add</button>`;
```

The compiler enforces this with clear errors:

```
Error: Side effect in template expression. Mutations must be in event handlers.
  → components/Counter.ts:5  ${count++}
```

### 7.4 List Rendering

Standard `.map()` works. The compiler optimizes it with keyed reconciliation:

```typescript
html`
  <ul>
    ${items.map((item) => html` <li key=${item.id}>${item.name}</li> `)}
  </ul>
`;
```

The `key` attribute is stripped by the compiler and used for efficient reconciliation. Additions, removals, and reorders produce minimal DOM mutations.

Missing `key` on a reactive array triggers a compiler warning and falls back to index-based reconciliation.

### 7.5 Conditional Rendering

Standard JavaScript conditionals work:

```typescript
html`
  ${isLoading
    ? html`<div class="spinner">Loading...</div>`
    : html`<div class="content">${data}</div>`}
`;
```

The compiler generates code that swaps DOM subtrees when the condition changes. The inactive branch is not evaluated. Swapping branches disposes the old branch's owner scope (cleaning up all subscriptions).

## 8. Ownership & Disposal

The compiler generates an ownership tree that mirrors the DOM tree. Every component creates an owner scope. Every signal subscription, `effect`, and `watch` belongs to an owner. Disposal is automatic and recursive.

### 8.1 Rules

- Unmount a component → owner scope disposes → all subscriptions, `effect`s, and `watch`es disposed.
- Conditional branch swaps → old branch's owner scope disposes.
- List item removed → that item's owner scope disposes.
- HMR swap → old component's owner scope disposes, new one creates fresh.

Memory leaks from undisposed subscriptions are framework bugs, not developer bugs. The developer never manually subscribes or unsubscribes.

### 8.2 Explicit Owner API

For cases where the runtime or compiler need explicit ownership scopes (conditional branches, list items, component mounting):

```typescript
import { createOwner, runWithOwner } from '@lattice/signals'

const owner = createOwner()

runWithOwner(owner, () => {
  // Effects and computeds created here are owned by this owner
  effect(() => { ... })
})

owner.dispose()  // Disposes everything created inside
```

`createOwner` creates a scope. Effects and computeds created inside `runWithOwner` are owned by that owner. Disposing the owner disposes everything created inside it. This is the same mechanism that implicit effect ownership uses, but explicit and decoupled from effects.

### 8.3 Shared State

Shared state (signals exported from modules) lives outside the ownership tree. Module-level signals persist for the lifetime of the application. This is correct — shared state should not be disposed when a component unmounts.

`effect` and `watch` created at module level persist for the application lifetime. `effect` and `watch` created inside components are owned by the component and disposed on unmount.

## 9. The Compiler

The Lattice compiler is a standalone package (`@lattice/compiler`) — a pure code transformer. TypeScript in, optimized JavaScript out.

### 9.1 What the Compiler Does

1. **Signal transform:** Identifies variables initialized with `signal()` or `computed()`. Rewrites all reads and writes of those variables to go through `.value`. Follows the edge case rules defined in Section 2.4.

2. **Template compilation:** Parses `html` tagged templates. Generates direct DOM construction code with signal subscriptions. No runtime template parsing.

3. **Dependency visibility:** Analyzes `effect()` and `watch()` bodies. Annotates discovered signal dependencies for IDE integration and dev tool display.

4. **Template purity enforcement:** Detects side effects in template expressions. Emits clear errors.

5. **CSS extraction:** Parses `css` tagged templates. Scopes selectors. Emits static CSS files.

6. **Source maps:** Full source map generation. Errors point to original TypeScript. Non-negotiable.

### 9.2 What the Compiler Does NOT Do

- It does NOT make plain `let` variables reactive. Only `signal()` and `computed()` are transformed.
- It does NOT analyze functions transitively. If you call `someFunction(count)`, the compiler passes the current value — it doesn't try to determine if `someFunction` needs a reactive binding.
- It does NOT transform object literals containing signal bindings into getter/setter objects (v1). Signal bindings in object literals become snapshots (Section 2.4).
- It does NOT process files that don't import from `lattice`. Plain TypeScript files are untouched.

### 9.3 Compile-Time Validation


## 11. Full Example — TodoMVC

```typescript
// stores/todos.ts
import { signal, computed } from 'lattice';

interface Todo {
  id: number;
  text: string;
  done: boolean;
}

export let todos = signal<Todo[]>([]);
export let filter = signal<'all' | 'active' | 'done'>('all');
export let editingId = signal<number | null>(null);

export let visible = computed(() => {
  if (filter === 'all') return todos;
  return todos.filter((t) => (filter === 'done' ? t.done : !t.done));
});

export let remaining = computed(() => todos.filter((t) => !t.done).length);

export let allDone = computed(() => todos.length > 0 && todos.every((t) => t.done));

export function add(text: string) {
  if (!text.trim()) return;
  todos.push({ id: Date.now(), text: text.trim(), done: false });
}

export function toggle(id: number) {
  const todo = todos.find((t) => t.id === id);
  if (todo) todo.done = !todo.done;
}

export function remove(id: number) {
  const idx = todos.findIndex((t) => t.id === id);
  if (idx >= 0) todos.splice(idx, 1);
}

export function edit(id: number, text: string) {
  const todo = todos.find((t) => t.id === id);
  if (todo) {
    todo.text = text.trim();
    editingId = null;
  }
}

export function toggleAll() {
  const target = !allDone;
  todos.forEach((t) => (t.done = target));
}

export function clearDone() {
  todos = todos.filter((t) => !t.done);
}
```

```typescript
  `;
}
```

### 17.1 Package API Boundaries

`@lattice/signals` is the raw reactive engine. It exports `signal`, `computed`, `effect`, `batch`, `untracked`, `createOwner`, `runWithOwner`, and the public types. The `effect` primitive here is the auto-tracked side effect with immediate execution.

`@lattice/runtime` (or the `lattice` re-export package) builds on top of `@lattice/signals` to provide `watch` (explicit source, old/new values, no initial run) and any other higher-level primitives. `watch` is a genuinely different primitive from `effect` — it has different semantics and is implemented separately, but built on the same signal engine internals.

The developer imports from `lattice` and gets everything. They never need to know which sub-package a primitive comes from.

**Done when:** All tests pass, public API is minimal and intentional, performance baselines established, package builds and the vanilla demo works.

### Phase 2: Deep Reactive Proxy (`@lattice/signals`)

Recursive Proxy wrapping backed by Phase 1 signals. Collection-level tracking strategy (Section 5.3). Array method interception. Identity stability. View-based element access.

**Key design:** Single reactive store per signal, path-based signal tracking, lightweight views for nested access, automatic granularity selection (fine-grained for individual access, coarse for iteration).

**Done when:** All mutation patterns (nested assignment, array methods, delete) are reactive and tested. No double-proxying. No memory leaks. Performance competitive with naive per-object proxying on small collections, significantly better on large collections (1k+ items).

### Phase 3: Compiler — Signal Transform (`@lattice/compiler`)

Parse TypeScript files. Identify `signal()` and `computed()` initializations. Rewrite all reads and writes of those bindings to use `.value`. Implement edge case rules from Section 2.4. Source map generation.

**Done when:** `let count = signal(0); count++` compiles to correct signal operations. Object shorthand captures values. Destructuring is not transformed. Factory returns require explicit accessors. Cross-module signal imports are recognized. Source maps are accurate.

### Phase 4: Compiler — Template Pass-Through (`@lattice/compiler`)

Parse `html` tagged templates. Generate runtime DOM creation code (pass-through — not yet optimized). Vite plugin wrapper.

**Done when:** A simple component renders in the browser. Template output matches expected DOM. Vite dev server runs with HMR.

### Phase 5: Compiler — Template Optimization (`@lattice/compiler`)

Generate direct signal-to-DOM subscriptions instead of effects. Static analysis to separate static vs dynamic parts. Keyed list reconciliation. Conditional branch optimization. CSS extraction and scoping.

**Done when:** Only affected DOM nodes update. Template purity violations produce compiler errors. Keyed lists handle add/remove/reorder efficiently. CSS is extracted and scoped at build time. Performance is competitive with Solid on js-framework-benchmark.

### Phase 6: Runtime (`@lattice/runtime`)

Component mounting, lifecycle hooks (`onMount`, `onDestroy`), ownership tree wired to DOM tree, `watch()` implementation, `mount()` API, component props.

**Done when:** Full TodoMVC app runs. Components mount, update, and unmount correctly. Ownership disposal cleans up all subscriptions. HMR works.

### Phase 7: Dev Tools & CLI

Browser extension. CLI scaffolding (`lattice create`). Build validation. Effect dependency visualization.

**Done when:** Developers can inspect live signal values, see the dependency graph, see `effect()` dependencies, and diagnose update propagation issues.

### Phase 8: Documentation & Examples

Documentation site. TodoMVC. Real-world example app. Migration guides.

**Done when:** A React developer can build a working app from the docs without external help.

### Phase 9+: Fullstack (Future)

SSR, hydration, resumability, islands, meta-framework. See Section 15 for architectural direction.

## 19. Open Design Questions

These are decisions that don't need to be made yet but should be tracked:

1. **Auto-batching in effects:** Should `effect()` callbacks automatically batch their internal signal writes? Currently effects don't auto-batch — writing to two signals inside an effect triggers two propagations. Auto-batching would make it one. Needs a decision and explicit test coverage either way.

2. **Template language server:** Tagged templates have worse IDE support than JSX (syntax highlighting, autocomplete, error squiggles). A language server plugin at least as good as lit-html's is critical for adoption. This is the single biggest adoption risk.

3. **Compiler auto-accessor for factory returns:** Section 2.4 specifies explicit getter/setter for factory returns in v1. Automatic transformation (detecting signal bindings in return objects and generating getter/setter) is desirable but complex. Track as a future compiler enhancement.

4. **`shallow` option implementation timing:** The `{ shallow: true }` API is designed in (Section 5.3) but may not be implemented in Phase 2 if the collection-level tracking strategy performs well enough. Benchmark first, then decide.

5. **`Map` and `Set` support:** Deep proxy needs to decide whether `Map` and `Set` are proxied (with method interception) or left as-is with a warning. Deferring to Phase 2 implementation.
