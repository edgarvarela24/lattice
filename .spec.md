# Lattice — Specification v1.5

> **Version:** 1.5.0-draft
> **Status:** Pre-implementation design spec
> **Date:** March 2026

---

## 1. What Lattice Is

Lattice is a **compiled TypeScript framework** built on fine-grained signals, deep reactive proxies, and compiled tagged templates. It combines Svelte's "signals as values" DX with Solid's fine-grained reactivity model — in plain `.ts` files, with no special file formats, no virtual DOM, and no `.value`.

The signal engine is platform-agnostic. Components render to DOM in the browser. The same reactive primitives work on the server, enabling a unified fullstack model for SSR, hydration, and resumability.

```typescript
import { signal, html } from 'lattice';

export function Counter() {
  let count = signal(0);

  return html` <button @click=${() => count++}>${count}</button> `;
}
```

You write `count++`. The compiler transforms it into a signal write. You write `${count}` in a template. The compiler generates a direct signal-to-DOM subscription at build time. One value changes, one DOM node updates. Nothing else runs.

### 1.1 Why Lattice Exists

**React** is simple to start but painful at scale: stale closures, `useEffect` footguns, a virtual DOM that diffs the entire tree when one value changes, and manual memoization that the React Compiler is only now beginning to automate.

**Svelte** proved that compiled signals with value semantics are the best DX in frontend. But reactivity only works in `.svelte` files, shared state requires `.svelte.ts` files, and the component format is a custom language — not plain TypeScript.

**Solid** proved that fine-grained signals with no virtual DOM deliver the best performance. But signals are containers with function-call getters (`count()` not `count`), destructuring breaks reactivity, and there's no compiler transform for signal DX.

**Lattice** takes the best of both: Svelte's "signals are values" DX powered by a compiler, Solid's fine-grained reactivity with no virtual DOM, deep reactive proxies so mutation works everywhere — all in plain TypeScript files, with a clear path to fullstack.

### 1.2 Core Principles

| Principle                       | Meaning                                                                                                        |
| ------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| **Signals are values**          | `count` is a number. You increment it with `++`. You pass it to functions. The compiler handles the rest.      |
| **Mutation works**              | `count++`, `todos.push(item)`, `user.name = 'x'` — all reactive, all tracked at the precise path that changed. |
| **One way to do things**        | `signal()` everywhere. Same syntax in components, shared modules, server code, anywhere. No special cases.     |
| **Plain TypeScript**            | No `.svelte`, no `.jsx`, no custom file formats. Components are functions in `.ts` files.                      |
| **Seven concepts**              | `signal`, `computed`, `effect`, `watch`, `html`, `toRaw`, `untracked`. That's the framework.                   |
| **Compiler-visible reactivity** | The compiler knows every dependency. It shows you, warns you, and catches mistakes.                            |
| **Platform-agnostic core**      | The signal engine runs anywhere JavaScript runs. The template layer targets the DOM. The boundary is clean.    |

### 1.3 The Public API

```typescript
import { signal, computed, effect, watch, html, toRaw, untracked } from 'lattice';
```

| Primitive                 | Purpose                                                                                                                                                                                                  |
| ------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `signal(value)`           | Reactive state. Compiler transforms reads/writes so it behaves as a plain value. Objects and arrays are deeply reactive via proxy — mutations at any depth are tracked at the precise path that changed. |
| `computed(fn)`            | Computed derivation. Lazy, cached, equality-checked. Re-evaluates when dependencies change.                                                                                                              |
| `effect(fn)`              | Reactive side effect. Auto-tracks dependencies. Compiler shows you what it depends on.                                                                                                                   |
| `watch(source, callback)` | Explicit side effect. Declared source, no auto-tracking in the callback. For when you need precise control.                                                                                              |
| `html`\`\`                | Compiled tagged template. Generates direct signal-to-DOM subscriptions at build time.                                                                                                                    |
| `toRaw(proxy)`            | Returns the underlying plain object from a reactive proxy. Use when passing state to third-party libraries that don't expect proxies.                                                                    |
| `untracked(fn)`           | Executes `fn` without tracking any signal reads inside it. Reads within `fn` do not become dependencies.                                                                                                 |

Seven words. Everything else is plain TypeScript.

### 1.4 Why These Names

The names are deliberately familiar. `signal` is the industry term (Solid, Angular, Preact). `computed` is what Vue, Solid, Angular, and MobX call it. `effect` is what Solid, Preact, and Angular call it. `watch` is what Vue calls it. `toRaw` is what Vue calls it. `untracked` is what Solid and Angular call it. Lattice's differentiator is the DX and the compiler, not the vocabulary. A developer coming from any modern framework already knows what these do.

## 2. Signals as Values

The core DX innovation: signals behave as plain values, not containers.

### 2.1 The Compiler Transform

The developer writes:

```typescript
let count = signal(0);
count++;
console.log(count);
```

The compiler outputs:

```typescript
import { signal } from '@lattice/signals';
const __count = signal(0);
__count.value++;
console.log(__count.value);
```

Every read of `count` becomes a `.value` read. Every write becomes a `.value` write. The developer never sees `.value`. The variable behaves as its contained type — `count` is a `number`, not a `Signal<number>`.

### 2.2 Where Signals Work

Signals work identically everywhere. No special files. No special scopes.

**In components:**

```typescript
import { signal, html } from 'lattice';

export function Counter() {
  let count = signal(0);
  return html` <button @click=${() => count++}>${count}</button> `;
}
```

**In shared modules:**

```typescript
// stores/counter.ts
import { signal } from 'lattice';

export let count = signal(0);
export function increment() {
  count++;
}
export function reset() {
  count = 0;
}
```

**In utility/logic files:**

```typescript
// composables/useDebounce.ts
import { signal, effect } from 'lattice';

export function useDebounce<T>(source: () => T, ms: number) {
  let debounced = signal(source());

  effect(() => {
    const val = source();
    const timeout = setTimeout(() => (debounced = val), ms);
    return () => clearTimeout(timeout);
  });

  return debounced;
}
```

**On the server:**

```typescript
// server/monitor.ts
import { signal, computed, effect } from 'lattice';

export let activeConnections = signal(0);
export let serverHealthy = computed(() => activeConnections < 1000);

effect(() => {
  if (!serverHealthy) {
    logger.warn(`High connection count: ${activeConnections}`);
  }
});
```

**Same pattern. Same syntax. Everywhere.**

### 2.3 What the Compiler Tracks

The compiler transforms reads and writes for any variable initialized with `signal()` or `computed()`. It does this by tracking the variable binding through its lexical scope:

- `let x = signal(0)` — `x` is a signal binding. All reads/writes transformed.
- `let y = computed(() => x * 2)` — `y` is a computed binding. Reads transformed (writes are an error).
- `let z = 5` — `z` is a plain variable. Not transformed. Not reactive.

The rule is explicit: **only variables initialized with `signal()` or `computed()` are transformed.** Everything else is plain TypeScript.

### 2.4 Compiler Edge Cases & Rules

The compiler's transform is based on one principle: **signal bindings are reactive through the variable name.** When a signal value leaves the variable binding, it becomes a snapshot.

#### Object Shorthand — Snapshot Semantics

```typescript
let count = signal(0);
const obj = { count }; // { count: 0 } — captures value, not binding

// Compiled:
const obj = { count: __count.value };
```

This is intentional. If `obj.count` were a live binding, objects passed to third-party libraries (`JSON.stringify(obj)`, `Object.keys(obj)`, etc.) would behave unexpectedly. The rule is simple: put a signal in an object, get a snapshot.

**If you want reactive objects, use `signal()` directly:**

```typescript
let state = signal({ count: 0 });
state.count++; // reactive — deep proxy handles this
```

#### Destructuring — Not Transformed

```typescript
let [a, b] = [signal(0), signal(1)]; // a and b are NOT signal bindings
```

Only variables directly initialized with `signal()` or `computed()` are transformed. Destructuring breaks the pattern. Write two declarations instead:

```typescript
let a = signal(0);
let b = signal(1);
```

#### Reassignment

```typescript
let count = signal(0);
count = 5; // → __count.value = 5 (signal write)
count = signal(10); // COMPILER ERROR: Cannot reassign signal binding to new signal
```

#### Factory Returns — Explicit Accessors (v1)

```typescript
// Returning signal bindings from a function captures values, not bindings:
function createCounter() {
  let count = signal(0);
  return { count }; // { count: 0 } — snapshot, not reactive
}

// To expose reactive state, use getter/setter:
function createCounter() {
  let count = signal(0);
  return {
    get count() {
      return count;
    },
    set count(v) {
      count = v;
    },
    increment() {
      count++;
    },
  };
}
```

This is verbose but explicit. The compiler may add automatic getter/setter generation for return objects containing signal bindings in a future version, but v1 uses explicit accessors to keep the compiler simple and predictable.

#### Closures in Async — Works Correctly

```typescript
let count = signal(0)
setTimeout(() => count++, 1000)     // ✅ works — closure captures __count
await fetch(...).then(() => count++) // ✅ works — same reason
```

The closure captures the `__count` signal object (a `const`), and `count++` inside the closure is transformed to `__count.value++`. Since `__count` never changes (only `.value` does), closures work correctly in all async contexts.

#### Exports — Cross-Module Signal Recognition

```typescript
// stores/counter.ts (compiled)
export let count = signal(0);

// Compiled output exports the signal object and metadata:
export { count$signal as count };
export const __lattice_meta = { count: 'signal' };

// components/Counter.ts (compiled)
import { count } from '../stores/counter';
// Compiler resolves the import, consults metadata, knows 'count' is a signal
// Transforms reads/writes accordingly
```

The importing file's compiler recognizes signal exports via the cross-module resolution protocol (see Section 9.4) and transforms reads/writes accordingly. Non-Lattice code importing a signal sees the raw signal object with `.value` — this is the interop escape hatch.

### 2.5 Passing Signals

When a signal variable is passed to a function, the compiler passes the current value:

```typescript
let count = signal(0);
console.log(count); // passes the value (number)
someFunction(count); // passes the value (number)
```

If you need to pass the signal itself (for composition), use an explicit accessor pattern:

```typescript
// Pass a getter — the function can read the signal reactively
useDebounce(() => count, 300);
```

This is the same pattern Vue, Solid, and Angular use for reactive composition. The accessor `() => count` is transformed by the compiler into `() => __count.value`, which creates a live binding.

### 2.6 Reading Without Tracking

`untracked()` executes a function without registering any signal reads as dependencies. This is useful inside `effect` or `computed` when you need to read a value without subscribing to it:

```typescript
import { signal, effect, untracked } from 'lattice';

let query = signal('');
let debugMode = signal(false);

effect(() => {
  // This effect should re-run when query changes, but NOT when debugMode changes
  const results = search(query);

  if (untracked(() => debugMode)) {
    console.log('Search results:', results);
  }
});
```

Without `untracked`, the effect would also re-run when `debugMode` changes. With it, `debugMode` is read but not tracked — the effect only depends on `query`.

The compiler transforms signal reads inside `untracked()` normally (to `.value` reads), but the signal engine's tracking stack is suspended for the duration of the call.

### 2.7 Extracting Raw Data

`toRaw()` returns the underlying plain JavaScript object from a reactive proxy, stripping all proxy wrapping recursively:

```typescript
import { signal, toRaw } from 'lattice';

let user = signal({ name: 'Alice', address: { city: 'NYC' } });

// Passing to a library that doesn't expect proxies
const raw = toRaw(user); // { name: 'Alice', address: { city: 'NYC' } }
JSON.stringify(raw); // works without triggering proxy traps
structuredClone(raw); // works without proxy interference
externalLibrary.process(raw); // no surprise Proxy behavior

// toRaw on a non-proxied value returns it unchanged
toRaw(42); // 42
toRaw('hello'); // 'hello'
```

**When to use `toRaw()`:**

- Passing state to third-party libraries (charting libs, form validators, serialization)
- Sending data over the network (`fetch` body, WebSocket messages)
- Storing snapshots (undo/redo history, localStorage)
- Any context where Proxy behavior could cause unexpected results

**When NOT to use `toRaw()`:**

- In templates — use the proxy directly (subscriptions are the point)
- In effects/computeds — use the proxy directly (tracking is the point)
- As a general practice — only reach for `toRaw` when interacting with code that doesn't know about proxies

Mutations on the raw object are NOT reactive. `toRaw()` gives you a snapshot for external consumption, not a backdoor for untracked writes.

## 3. Computed State

`computed()` creates a computed value that automatically updates when its dependencies change. It is lazy (only evaluates when read), cached (returns the cached value if dependencies haven't changed), and equality-checked (doesn't propagate if the new value equals the old value).

```typescript
import { signal, computed, html } from 'lattice';

export function TodoList() {
  let todos = signal<Todo[]>([]);
  let filter = signal<'all' | 'active' | 'done'>('all');

  let visible = computed(() =>
    filter === 'all' ? todos : todos.filter((t) => (filter === 'done' ? t.done : !t.done)),
  );

  let remaining = computed(() => todos.filter((t) => !t.done).length);

  return html`
    <p>${remaining} items left</p>
    <ul>
      ${visible.map((todo) => html` <li key=${todo.id}>${todo.text}</li> `)}
    </ul>
  `;
}
```

### 3.1 Computed Rules

- The function passed to `computed()` must be a pure computation. No side effects.
- `computed` bindings are read-only. Writing to a `computed` variable is a compiler error.
- `computed` uses dynamic dependency tracking internally (via the signal engine's observer system) to discover which signals the function reads.
- Custom equality can be provided: `computed(fn, { equals: (a, b) => a.id === b.id })`.

### 3.2 Error Handling in Computed

If a computed's function throws, the error is cached:

- First evaluation throws → error is cached. Subsequent `.value` reads re-throw the cached error.
- When a dependency changes, the computed re-evaluates. If the new evaluation succeeds, the error clears. If it throws again, the new error is cached.
- This avoids infinite re-evaluation of broken functions and naturally clears when inputs change.

### 3.3 Circular Dependency Detection

If computed A reads computed B which reads computed A, the runtime detects the re-entrant read and throws a clear error:

```
Error: Circular dependency detected in computed
```

Self-referencing computeds (a computed that reads itself) are also detected. After the error, the system is in a consistent state — no flags are corrupted.

## 4. Reactive Side Effects

Lattice provides two primitives for side effects: `effect` for auto-tracked reactivity, and `watch` for explicit source declaration. Both are tied to the ownership tree and automatically cleaned up when their scope disposes.

### 4.1 Effect

`effect()` is the primary side effect primitive. It executes immediately, auto-tracks which signals are read inside it, and re-executes when any of those signals change.

```typescript
import { signal, effect } from 'lattice';

let count = signal(0);

// Runs immediately, re-runs when count changes
effect(() => {
  document.title = `${count} items`;
});
```

**Cleanup:** Returning a function from `effect` registers a cleanup that executes before the next re-run or on disposal.

```typescript
effect(() => {
  const controller = new AbortController();

  fetch(`/api/search?q=${query}`, { signal: controller.signal })
    .then((res) => res.json())
    .then((data) => (results = data));

  return () => controller.abort();
});
```

**Common patterns:**

```typescript
// DOM sync
effect(() => {
  document.title = `${count} items`;
});

// Data fetching with cancellation
effect(() => {
  if (query.length < 3) {
    results = [];
    return;
  }

  const controller = new AbortController();
  loading = true;

  fetch(`/api/search?q=${query}`, { signal: controller.signal })
    .then((res) => res.json())
    .then((data) => {
      results = data;
      loading = false;
    })
    .catch((err) => {
      if (err.name !== 'AbortError') {
        error = err;
        loading = false;
      }
    });

  return () => controller.abort();
});

// Debouncing
effect(() => {
  const timeout = setTimeout(() => (debouncedQuery = query), 300);
  return () => clearTimeout(timeout);
});

// LocalStorage sync
effect(() => {
  localStorage.setItem('theme', theme);
});

// Polling
effect(() => {
  fetchPrice(symbol).then((p) => (price = p));
  const interval = setInterval(() => {
    fetchPrice(symbol).then((p) => (price = p));
  }, 5000);
  return () => clearInterval(interval);
});

// WebSocket
effect(() => {
  const ws = new WebSocket(`wss://api.example.com/chat/${room}`);
  ws.onopen = () => (connected = true);
  ws.onclose = () => (connected = false);
  ws.onmessage = (e) => messages.push(JSON.parse(e.data));
  return () => ws.close();
});
```

### 4.2 Compiler-Enhanced Dependency Visibility

The compiler knows which variables are signal bindings. When it sees signal reads inside an `effect()`, it can:

**Show dependencies in IDE/build output:**

```
// Compiler annotation:
// Dependencies: [query, isSearching]
effect(() => {
  if (isSearching) {
    fetchResults(query)
  }
})
```

**Warn about accidental dependencies:**

```
Warning: effect() reads signal 'userName' which appears to be a debug statement.
  If intentional, ignore. If not, use untracked() to read without tracking.
  → components/Search.ts:14  console.log(userName)
```

**Detect dependency changes between edits:**

```
Info: effect() dependency set changed. Previously: [query]. Now: [query, isSearching].
  → components/Search.ts:12
```

This transforms the traditional footgun of auto-tracked effects into a transparent, debuggable system. The compiler sees what the runtime tracks.

### 4.3 Watch

`watch()` is the explicit-source alternative. Use it when you need precise control over what triggers re-execution, or when you need access to old and new values.

```typescript
import { signal, watch } from 'lattice';

let query = signal('');

// Explicit source — only re-runs when query changes
watch(
  () => query,
  (newQuery, oldQuery) => {
    analytics.track('search_changed', { from: oldQuery, to: newQuery });
  },
);

// Multiple explicit sources
watch(
  () => [firstName, lastName],
  ([first, last]) => {
    analytics.track('name_changed', { first, last });
  },
);

// Cleanup
watch(
  () => query,
  (q) => {
    const timeout = setTimeout(() => search(q), 300);
    return () => clearTimeout(timeout);
  },
);
```

### 4.4 Effect vs Watch

|                         | `effect`                                                           | `watch`                                                                                                                              |
| ----------------------- | ------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------ |
| **Dependency tracking** | Automatic — reads inside the function are tracked                  | Explicit — you declare the source                                                                                                    |
| **Runs on init**        | Yes                                                                | No — runs on first change                                                                                                            |
| **Old/new values**      | No                                                                 | Yes — callback receives `(newValue, oldValue)`                                                                                       |
| **When to use**         | Default choice. DOM sync, data fetching, any reactive side effect. | When you need to react to a _specific_ signal change, need old/new comparison, or want to ignore other signals read in the callback. |

**Rule of thumb:** Start with `effect`. Use `watch` when you need its specific capabilities.

### 4.5 Effect Rules

- `effect` executes immediately on creation, then re-executes when tracked dependencies change.
- The function body is auto-tracked: any signal read inside it becomes a dependency.
- Returning a function registers cleanup that runs before re-execution or on disposal.
- `effect` is tied to the ownership tree. Component unmount → automatic cleanup.
- Signals written inside `effect` do not cause the same `effect` to re-trigger (no infinite loops).

### 4.6 Watch Rules

- The first argument is a **source function** — a getter that returns the value(s) to watch. The compiler transforms signal reads inside it.
- The second argument is a **callback** that runs when the source value changes. It receives the new and old values. The callback is NOT auto-tracked — signal reads in the callback do not become dependencies.
- `watch` does not run on initialization. It runs on the first change.
- Returning a function from the callback registers cleanup.
- `watch` is tied to the ownership tree.

## 5. Deep Reactivity

Lattice uses deep reactive proxies with path-level tracking. Mutation works at any depth, and only the specific path that changed is notified.

### 5.1 What Is Reactive

All of the following trigger precise, path-level updates:

```typescript
// Top-level assignment (compiler-transformed signal)
count = 5;
count++;

// Nested property assignment (deep proxy — notifies subscribers of user.name specifically)
state.user.name = 'Alice';
state.settings.notifications.email = false;

// Array methods (deep proxy — notifies collection-level subscribers)
state.items.push(newItem);
state.items.splice(2, 1);
state.items[3] = updatedItem;
state.items.sort((a, b) => a.name.localeCompare(b.name));

// Deletion
delete state.user.middleName;
```

### 5.2 How It Works

When a `signal()` is initialized with an object or array, the value is wrapped in a recursive Proxy. Each property access path maps to a signal internally. Reading a path subscribes the reader to that specific path. Writing a path notifies only subscribers of that path and any parent paths.

```typescript
let user = signal({ name: 'Alice', address: { city: 'NYC' } });

user.name = 'Bob'; // notifies subscribers of user.name and user
user.address.city = 'LA'; // notifies subscribers of user.address.city, user.address, and user
```

This is what makes Lattice's reactivity precise. An effect reading `user.name` does not re-run when `user.address.city` changes. An effect reading `user.address.city` does not re-run when `user.name` changes. Each consumer is subscribed only to the exact paths it reads.

### 5.3 Why Deep Path Tracking Matters

Consider a dashboard with independent panels:

```typescript
let dashboard = signal({
  users: { online: 142, total: 8830 },
  revenue: { today: 4200, mtd: 89000 },
  system: { cpu: 0.45, memory: 0.72, errorRate: 0.001 },
});

// This effect only re-runs when system metrics change:
effect(() => {
  document.documentElement.dataset.theme = dashboard.system.cpu > 0.9 ? 'danger' : 'normal';
});

// A revenue update does NOT trigger the effect above.
// No computed intermediary needed. No defensive wrapping. It just works.
dashboard.revenue.today = 4500; // system effect is not notified
```

Without path tracking, every consumer of `dashboard` would re-run on every mutation anywhere in the object. Developers would need to add `computed` wrappers defensively around every access to prevent over-notification. Path tracking eliminates this entire category of boilerplate.

### 5.4 Path Tracking Strategy

The top-level `signal()` creates a single **reactive store** that tracks all access paths. Path-based signals are created lazily on first access. The store holds the raw (unproxied) data and a map of path-based signals.

When a nested object or array element is accessed, the store returns a lightweight **view** — a Proxy that routes reads and writes back through the store's centralized tracking.

**Two granularities, chosen automatically:**

| Access Pattern                 | Tracking Level            | Example                                               |
| ------------------------------ | ------------------------- | ----------------------------------------------------- |
| **Individual property access** | Fine-grained (path-based) | `todos[5].done` → tracks path `todos.5.done`          |
| **Iteration**                  | Coarse (collection-level) | `todos.filter(...)` → tracks `todos` collection       |
| **Nested object property**     | Fine-grained (path-based) | `user.address.city` → tracks path `user.address.city` |

When `.filter()`, `.map()`, `.forEach()`, or other iteration methods are called on a reactive array, the store registers a single "collection iterated" dependency rather than N individual element dependencies.

This means:

- `todos.filter(t => !t.done)` registers **1 dependency** (collection-level), not 10,000
- `todos[5].done = true` triggers the collection dependency (the filter must re-run — it reads all elements) AND the specific path dependency (a component reading only `todos[5].done` also updates)
- Individual element access (`todos[5].done`) gets fine-grained tracking. A `computed` that only reads one specific element won't re-run when a different element changes.

The developer never thinks about which tracking mode is active.

### 5.5 Reactive References Across Boundaries

Objects accessed through the proxy remain reactive when passed to functions or child components. The proxy travels with the reference:

```typescript
let app = signal({
  user: { name: 'Alice', prefs: { theme: 'dark' } },
});

// Helper function — receives a proxied object, mutations are reactive
function resetUser(user: User) {
  user.name = ''; // reactive — notifies subscribers of the path
  user.prefs.theme = 'light'; // reactive — notifies subscribers of the nested path
}
resetUser(app.user); // works — app.user returns a view proxy

// Child component — receives a proxied object, reads are tracked
function UserCard({ user }: { user: User }) {
  return html`
    <span>${user.name}</span>
    <!-- subscribes to user.name specifically -->
  `;
}
```

This is the core DX advantage of deep reactive proxies: **reactivity is a property of the data, not a property of how you access it.** It doesn't matter whether the mutation happens at the top level, inside a helper function, in a child component, or in a callback. The proxy intercepts it.

### 5.6 What Array Methods Return Through the Proxy

Array methods that create new arrays (`filter`, `map`, `slice`, `concat`, `flat`, `flatMap`) return **plain, non-proxied arrays**. The returned array is a new value, not a view into the reactive store:

```typescript
let todos = signal<Todo[]>([
  { id: 1, text: 'Buy milk', done: false },
  { id: 2, text: 'Walk dog', done: true },
]);

// filter returns a plain array — NOT reactive
const active = todos.filter((t) => !t.done);
active[0].done = true; // ❌ mutates the raw object, NOT the store
// this is NOT reactive, no subscribers notified

// To reactively toggle, go through the original signal:
todos[0].done = true; // ✅ reactive — subscribers of todos.0.done notified
```

This is consistent and predictable: **the proxy wraps the signal's value. New arrays aren't the signal's value.**

Array methods that mutate in place (`push`, `pop`, `shift`, `unshift`, `splice`, `sort`, `reverse`, `fill`, `copyWithin`) operate on the proxied array and trigger reactive notifications:

```typescript
todos.push({ id: 3, text: 'New todo', done: false }); // ✅ reactive
todos.splice(1, 1); // ✅ reactive
todos.sort((a, b) => a.id - b.id); // ✅ reactive
```

Reassigning the signal to a new array (e.g., from a `filter` result) is also reactive — it replaces the entire value:

```typescript
export function clearDone() {
  todos = todos.filter((t) => !t.done);
  // filter creates a plain array → assigned to todos → new array gets proxied
  // all subscribers of todos are notified
}
```

### 5.7 Structural Array Mutations and Path Invalidation

When an array undergoes a structural mutation — one that changes the indices of existing elements — all index-based path subscribers on that array are invalidated:

```typescript
let items = signal([
  { name: 'A' }, // index 0
  { name: 'B' }, // index 1
  { name: 'C' }, // index 2
  { name: 'D' }, // index 3
]);

// A computed that reads one specific element:
let thirdName = computed(() => items[2].name); // subscribes to path items.2.name

// Splice removes index 1:
items.splice(1, 1);
// items is now: [{ name: 'A' }, { name: 'C' }, { name: 'D' }]
// items[2].name is now 'D', not 'C'

// thirdName MUST re-evaluate — the value at its path changed.
// The proxy invalidates all index-based subscribers on structural mutation.
```

**Structural mutations** are: `splice`, `shift`, `unshift`, `sort`, `reverse`, and direct length assignment. These invalidate all indexed path subscribers on the array.

**Non-structural mutations** are: setting a specific index (`items[5] = x`), mutating a property on an element (`items[5].done = true`), `push` (only adds to the end — existing indices don't shift), `pop` (only removes from the end), `fill`, `copyWithin`. These notify only the specific affected paths.

This matches developer expectations: if you sort an array, everything reading from it should update. If you toggle one property on one element, only that property's subscribers should update.

### 5.8 Identity Semantics

Views maintain stable identity:

```typescript
const a = user.address; // returns view
const b = user.address; // returns same view
a === b; // true
```

The same object is never double-proxied. Non-extensible and frozen objects are not proxied (they can't be reactive — the compiler warns if detected).

### 5.9 Shallow Escape Hatch

For performance-critical cases where even path-level tracking overhead is too much:

```typescript
let items = signal<Item[]>([], { shallow: true });
// Array itself is reactive (push/splice/assignment trigger updates)
// Individual item properties are NOT tracked through the proxy
// items[0].done = true → NOT tracked (use with caution)
```

This is designed into the API from day one to avoid breaking changes later, even if not implemented immediately.

### 5.10 Batching

Multiple synchronous mutations batch into one propagation:

```typescript
import { batch } from 'lattice';

batch(() => {
  theme = 'dark';
  sidebarOpen = false;
  activeTab = 'logs';
});
// → ONE propagation, not three
```

Batching also occurs automatically within event handlers and `effect`/`watch` callbacks.

### 5.11 Glitch-Free

The signal graph ensures computed values never see inconsistent intermediate states. If signal `s` feeds both `computed A` and `computed B`, and an `effect` reads both, it sees consistent values from a single propagation — never a mix of old and new.

The current implementation achieves this through eager evaluation in computed's `notify` — when a dependency changes, an observed computed immediately re-evaluates so downstream effects see fresh values. A future optimization could use topological sorting (processing nodes in dependency-depth order) to avoid unnecessary intermediate evaluations in deep computed chains, but the current approach is correct and simpler.

## 6. Components

Components are functions that return an `html` tagged template. No classes. No decorators. No special file format.

### 6.1 Basic Component

```typescript
import { signal, html } from 'lattice';

export function Counter() {
  let count = signal(0);

  return html` <button @click=${() => count++}>${count}</button> `;
}
```

### 6.2 Props

Components receive props as function arguments:

```typescript
import { signal, html } from 'lattice';

interface CounterProps {
  initial?: number;
  label?: string;
}

export function Counter({ initial = 0, label = 'Count' }: CounterProps = {}) {
  let count = signal(initial);

  return html` <button @click=${() => count++}>${label}: ${count}</button> `;
}
```

Using a component:

```typescript
import { html } from 'lattice';
import { Counter } from './Counter';

export function App() {
  return html`
    <div>
      <${Counter} initial=${5} label=${'Clicks'} />
      <${Counter} />
    </div>
  `;
}
```

### 6.3 Children / Slots

Components can accept children:

```typescript
import { html } from 'lattice';

export function Card({ title }: { title: string }) {
  return html`
    <div class="card">
      <h2>${title}</h2>
      <div class="card-body">
        <slot />
      </div>
    </div>
  `;
}

// Usage
html`
  <${Card} title=${'Hello'}>
    <p>This goes in the slot.</p>
  </${Card}>
`;
```

### 6.4 Lifecycle

Components have two lifecycle hooks, available as imports:

```typescript
import { signal, html, onMount, onDestroy } from 'lattice';

export function Timer() {
  let elapsed = signal(0);
  let interval: number;

  onMount(() => {
    interval = setInterval(() => elapsed++, 1000);
  });

  onDestroy(() => {
    clearInterval(interval);
  });

  return html`<p>Elapsed: ${elapsed}s</p>`;
}
```

- `onMount(fn)` — runs after the component's DOM is created and inserted. The function can return a cleanup function (equivalent to also registering `onDestroy`).
- `onDestroy(fn)` — runs when the component is removed from the DOM. All signal subscriptions and `effect`/`watch` calls are automatically cleaned up by the ownership system — `onDestroy` is only needed for non-Lattice resources (timers, event listeners, websockets, etc).

### 6.5 Scoped Styles

The `css` tagged template provides component-scoped styles:

```typescript
import { signal, html, css } from 'lattice';

const styles = css`
  button {
    background: var(--color-primary);
    color: white;
    padding: 8px 16px;
    border-radius: 4px;
  }
  button:hover {
    background: var(--color-primary-dark);
  }
`;

export function Counter() {
  let count = signal(0);

  return html` <button @click=${() => count++}>${count}</button> `;
}
```

The compiler:

1. Extracts CSS at build time (no runtime injection).
2. Scopes selectors with a unique component attribute (e.g., `button[data-l-counter]`).
3. Adds the scoping attribute to the component's root elements.
4. Emits static CSS files that the browser caches normally.

## 7. Templates

Templates use `html` tagged template literals — standard JavaScript. The compiler parses them at build time and emits optimized DOM operations.

### 7.1 What the Compiler Generates

```typescript
// You write:
html`
  <div class=${theme}>
    <span>${count}</span>
    <p>${name}</p>
  </div>
`;

// Compiler generates (conceptually):
function template() {
  const div = createElement('div');
  const span = createElement('span');
  const p = createElement('p');
  div.append(span, p);

  // Direct subscriptions — no effect wrappers
  __theme.subscribe((val) => (div.className = val));
  __count.subscribe((val) => (span.textContent = val));
  __name.subscribe((val) => (p.textContent = val));

  return div;
}
```

Each DOM node subscribes to exactly the signals it reads. When `count` changes, only the span updates. The div and p are untouched. No virtual DOM. No diffing.

For template expressions that involve multiple signals, the compiler generates a `computed` and subscribes to that:

```typescript
// You write:
html`<span>${firstName} ${lastName}</span>`;

// Compiler generates:
const __expr = computed(() => `${__firstName.value} ${__lastName.value}`);
__expr.subscribe((val) => (span.textContent = val));
```

### 7.2 Template Bindings

```typescript
// Text interpolation
html`<span>${count}</span>`;

// Attribute binding
html`<div class=${theme}></div>`;

// Boolean attribute
html`<button ?disabled=${isLoading}>Submit</button>`;

// Property binding
html`<input .value=${query} />`;

// Event handler
html`<button @click=${() => count++}>Click</button>`;

// Component rendering
html`<${Counter} initial=${5} />`;
```

### 7.3 Template Purity

Template expressions must be pure reads. Mutations live in event handlers.

```typescript
// ✅ Pure reads — allowed in template expressions
html`<span>${count}</span>`;
html`<span>${count * 2}</span>`;
html`<span>${formatDate(createdAt)}</span>`;
html`<span>${items.length} items</span>`;

// ❌ Side effects — compiler errors
html`<span>${count++}</span>`;
html`<span>${items.push(x)}</span>`;

// ✅ Mutations in event handlers — always allowed
html`<button @click=${() => count++}>Click</button>`;
html`<button @click=${() => items.push(newItem)}>Add</button>`;
```

The compiler enforces this with clear errors:

```
Error: Side effect in template expression. Mutations must be in event handlers.
  → components/Counter.ts:5  ${count++}
```

### 7.4 List Rendering

Standard `.map()` works. The compiler optimizes it with keyed reconciliation:

```typescript
html`
  <ul>
    ${items.map((item) => html` <li key=${item.id}>${item.name}</li> `)}
  </ul>
`;
```

The `key` attribute is stripped by the compiler and used for efficient reconciliation. Additions, removals, and reorders produce minimal DOM mutations.

Missing `key` on a reactive array triggers a compiler warning and falls back to index-based reconciliation.

### 7.5 Conditional Rendering

Standard JavaScript conditionals work:

```typescript
html`
  ${isLoading
    ? html`<div class="spinner">Loading...</div>`
    : html`<div class="content">${data}</div>`}
`;
```

The compiler generates code that swaps DOM subtrees when the condition changes. The inactive branch is not evaluated. Swapping branches disposes the old branch's owner scope (cleaning up all subscriptions).

## 8. Ownership & Disposal

The compiler generates an ownership tree that mirrors the DOM tree. Every component creates an owner scope. Every signal subscription, `effect`, and `watch` belongs to an owner. Disposal is automatic and recursive.

### 8.1 Rules

- Unmount a component → owner scope disposes → all subscriptions, `effect`s, and `watch`es disposed.
- Conditional branch swaps → old branch's owner scope disposes.
- List item removed → that item's owner scope disposes.
- HMR swap → old component's owner scope disposes, new one creates fresh.

Memory leaks from undisposed subscriptions are framework bugs, not developer bugs. The developer never manually subscribes or unsubscribes.

### 8.2 Explicit Owner API

For cases where the runtime or compiler need explicit ownership scopes (conditional branches, list items, component mounting):

```typescript
import { createOwner, runWithOwner } from '@lattice/signals'

const owner = createOwner()

runWithOwner(owner, () => {
  // Effects and computeds created here are owned by this owner
  effect(() => { ... })
})

owner.dispose()  // Disposes everything created inside
```

`createOwner` creates a scope. Effects and computeds created inside `runWithOwner` are owned by that owner. Disposing the owner disposes everything created inside it. This is the same mechanism that implicit effect ownership uses, but explicit and decoupled from effects.

### 8.3 Shared State

Shared state (signals exported from modules) lives outside the ownership tree. Module-level signals persist for the lifetime of the application. This is correct — shared state should not be disposed when a component unmounts.

`effect` and `watch` created at module level persist for the application lifetime. `effect` and `watch` created inside components are owned by the component and disposed on unmount.

## 9. The Compiler

The Lattice compiler is a standalone package (`@lattice/compiler`) — a pure code transformer. TypeScript in, optimized JavaScript out.

### 9.1 What the Compiler Does

1. **Signal transform:** Identifies variables initialized with `signal()` or `computed()`. Rewrites all reads and writes of those variables to go through `.value`. Follows the edge case rules defined in Section 2.4.
2. **Cross-module signal resolution:** Recognizes signal imports from other compiled modules via the metadata protocol (Section 9.4). Transforms reads/writes of imported signal bindings.
3. **Template compilation:** Parses `html` tagged templates. Generates direct DOM construction code with signal subscriptions. No runtime template parsing.
4. **Dependency visibility:** Analyzes `effect()` and `watch()` bodies. Annotates discovered signal dependencies for IDE integration and dev tool display.
5. **Template purity enforcement:** Detects side effects in template expressions. Emits clear errors.
6. **CSS extraction:** Parses `css` tagged templates. Scopes selectors. Emits static CSS files.
7. **Source maps:** Full source map generation. Errors point to original TypeScript. Non-negotiable.

### 9.2 What the Compiler Does NOT Do

- It does NOT make plain `let` variables reactive. Only `signal()` and `computed()` are transformed.
- It does NOT analyze functions transitively. If you call `someFunction(count)`, the compiler passes the current value — it doesn't try to determine if `someFunction` needs a reactive binding.
- It does NOT transform object literals containing signal bindings into getter/setter objects (v1). Signal bindings in object literals become snapshots (Section 2.4).
- It does NOT process files that don't import from `lattice`. Plain TypeScript files are untouched.

### 9.3 Compile-Time Validation

```
Error: Side effect in template expression. Mutations must be in event handlers.
  → components/Counter.ts:5  ${count++}

Error: Cannot assign to computed binding 'doubled'. Computed values are read-only.
  → stores/counter.ts:8  doubled = 10

Error: Cannot reassign signal binding 'count' to a new signal. Did you mean: count = 10
  → stores/counter.ts:5  count = signal(10)

Error: @clck is not a valid event binding. Did you mean @click?
  → components/Counter.ts:4

Warning: Reactive array mapped without key. Add key=${...} for efficient updates.
  → components/TodoList.ts:8

Warning: effect() reads signal 'debugName' which may be an unintended dependency.
  → components/Search.ts:14

Warning: Expression never changes. Use a plain attribute instead.
  → components/Layout.ts:3  class=${"static-class"}
```

### 9.4 Cross-Module Signal Resolution

When a file imports from another Lattice module, the compiler needs to know which imports are signal or computed bindings. This is handled through a two-tier resolution protocol.

#### How It Works

**When compiling an exporting file**, the compiler identifies all `signal()` and `computed()` exports and emits a `__lattice_meta` metadata export alongside the compiled code:

```typescript
// SOURCE: stores/counter.ts
import { signal, computed } from 'lattice';

export let count = signal(0);
export let doubled = computed(() => count * 2);
export const MAX = 100;
export function increment() {
  count++;
}

// COMPILED OUTPUT: stores/counter.ts
import { signal, computed } from '@lattice/signals';

const count$signal = signal(0);
const doubled$computed = computed(() => count$signal.value * 2);
export const MAX = 100;
export function increment() {
  count$signal.value++;
}

export { count$signal as count, doubled$computed as doubled };
export const __lattice_meta = { count: 'signal', doubled: 'computed' };
```

**When compiling an importing file**, the compiler resolves each import and checks whether the source module has signal metadata:

```typescript
// SOURCE: components/Display.ts
import { count, doubled, MAX } from '../stores/counter';

// Compiler resolves the import:
// - count → signal (from metadata) → transform reads/writes to .value
// - doubled → computed (from metadata) → transform reads to .value
// - MAX → not in metadata → leave as plain value
```

#### Two-Tier Lookup

**Tier 1 — In-memory cache (local files):** The compiler plugin maintains a `Map<moduleId, Record<string, 'signal' | 'computed'>>` that is populated during the current build as files are transformed. When processing imports from local files, this map is consulted. This handles all files within the project with zero disk I/O.

Because both Vite's dev server and Rollup's production build resolve module graphs depth-first (dependencies are transformed before dependents), the metadata for any local import is always available when needed. **This is single-pass in both dev and production.**

**Tier 2 — Read from dependency's reactive graph (npm packages):** When tier 1 misses (the import is from `node_modules` or another pre-compiled source), the compiler reads the dependency's reactive graph file (see Section 15). This is cached after first read. This enables publishing and consuming Lattice libraries — a library author builds their package, the reactive graph ships with the compiled output, and consumers' compilers discover signal metadata automatically. For backward compatibility, the compiler falls back to detecting `__lattice_meta` exports in compiled JavaScript if no graph file is found.

On dev server startup, the project-level reactive graph file (Section 15) pre-populates the tier 1 cache, eliminating cold-start penalties.

#### Metadata Stripping

The `__lattice_meta` export is a backward-compatibility fallback for cross-module resolution. The preferred mechanism is the reactive graph file (Section 15), which provides richer information and cleaner separation from runtime code. The compiler strips `__lattice_meta` from the final bundle output so it never reaches the browser:

```typescript
// During output generation, __lattice_meta exports are removed.
// Zero runtime cost.
```

#### Non-Lattice Consumers

If a non-Lattice project imports from a compiled Lattice module, they get raw signal objects. The `__lattice_meta` export is an inert object they never touch:

```typescript
// React project importing from a Lattice library:
import { count } from '@acme/ui-state';

count.value; // works — signal object with .value
count.value++; // works — but no compiler sugar
```

This is the interop escape hatch: `.value` is always available for direct access.

### 9.5 Compiler Architecture — Rust + OXC

The Lattice compiler is written in **Rust**, built on the **OXC** (oxc-project) toolchain. It is exposed to Node.js via NAPI bindings for integration with the Vite plugin.

#### Why Rust + OXC

Compiler speed directly impacts developer experience — every file save triggers a recompile during development. The compiler must be fast enough that HMR feels instant (sub-10ms per file). Rust + OXC is the fastest viable option:

- **OXC's parser** is 3x faster than SWC and passes all Test262 stage 4 tests. It is production-ready and used by Rolldown (Vite's future bundler).
- **OXC's traverse system** provides a typed visitor pattern (`Traverse` trait) with parent context access, efficient tree navigation, and Rust's memory safety guarantees.
- **OXC's codegen** handles JavaScript output with source map generation.
- **OXC's AST** uses distinct types for different identifier roles (`BindingIdentifier`, `IdentifierReference`, `IdentifierName`), which maps directly to the compiler's need to distinguish declarations from reads.
- **OXC is the foundation of Vite's future toolchain** (via Rolldown). Building on OXC means the Lattice compiler is native to the same ecosystem as its build tool.

#### Crate Dependencies

```toml
[dependencies]
oxc_parser     = "0.114"    # Parse TS/JS/JSX/TSX → AST
oxc_ast        = "0.114"    # AST type definitions
oxc_traverse   = "0.114"    # AST traversal with parent context
oxc_codegen    = "0.114"    # AST → JavaScript + source maps
oxc_allocator  = "0.114"    # Arena allocator (fast AST allocation)
oxc_semantic   = "0.114"    # Scope analysis, binding resolution
oxc_span       = "0.114"    # Source positions for source maps
napi           = "2"        # Node.js native bindings
napi-derive    = "2"        # NAPI proc macros
```

#### Compiler Pipeline

```
Source (.ts)
  → oxc_parser (parse to AST)
  → oxc_semantic (scope analysis, binding resolution)
  → SignalTransform (Lattice — rewrite signal bindings to .value)
  → TemplateTransform (Lattice — compile html`` to DOM operations)
  → MetadataEmitter (Lattice — emit __lattice_meta for exports)
  → CssExtractor (Lattice — extract and scope css`` templates)
  → oxc_codegen (AST → JavaScript + source maps)
```

Every step except the four Lattice transforms is provided by OXC crates. The Lattice-specific transforms implement OXC's `Traverse` trait:


#### Template Parser

OXC parses TypeScript but not HTML inside tagged template literals. The template parser is a custom Rust module within the compiler crate. It handles a restricted subset of HTML:

- Finds `html` tagged template expressions in the AST
- Extracts template literal strings and expression holes
- Parses the HTML structure (elements, attributes, text nodes, self-closing tags)
- Produces a template AST that the codegen turns into DOM construction code

This is simpler than a full HTML5 parser — no `<script>`, no document mode, no error recovery for malformed HTML. The templates are well-formed component markup with known expression boundaries.

#### NAPI Bridge

The Rust compiler is exposed to Node.js via NAPI (native bindings), following the same pattern OXC uses for its own npm packages:

```typescript
// @lattice/compiler — thin Node.js wrapper around the Rust crate
import { transformSync } from '@lattice/compiler-napi';

// Returns: { code: string, sourceMap: string, signalExports: Record<string, string> | null }
const result = transformSync(sourceCode, {
  filename: 'Counter.ts',
  importedSignals: { '../stores/counter': { count: 'signal', doubled: 'computed' } },
});
```

The Vite plugin calls `transformSync` for each file. The cross-module metadata cache lives in JavaScript (Section 9.4). The Rust side does all parsing, transformation, and code generation. The boundary is clean: string in, string + metadata out.

```typescript
// @lattice/vite — Vite plugin
export default function latticePlugin() {
  const signalMeta = new Map<string, Record<string, string>>();

  return {
    name: 'lattice',
    transform(code: string, id: string) {
      if (!hasLatticeImport(code)) return null;

      const result = transformSync(code, {
        filename: id,
        importedSignals: resolveSignalImports(id, signalMeta),
      });

      if (result.signalExports) {
        signalMeta.set(id, result.signalExports);
      }

      return { code: result.code, map: result.sourceMap };
    },
  };
}
```

#### Build and Distribution

The compiler crate builds to platform-specific native modules via `@napi-rs/cli`. Published npm packages include pre-built binaries for major platforms (macOS arm64/x64, Linux x64/arm64, Windows x64) with a WASM fallback for unsupported platforms. Developers install the compiler as a normal npm dependency — no Rust toolchain required for end users.

## 11. Full Example — TodoMVC

```typescript
// stores/todos.ts
import { signal, computed } from 'lattice';

interface Todo {
  id: number;
  text: string;
  done: boolean;
}

export let todos = signal<Todo[]>([]);
export let filter = signal<'all' | 'active' | 'done'>('all');
export let editingId = signal<number | null>(null);

export let visible = computed(() => {
  if (filter === 'all') return todos;
  return todos.filter((t) => (filter === 'done' ? t.done : !t.done));
});

export let remaining = computed(() => todos.filter((t) => !t.done).length);

export let allDone = computed(() => todos.length > 0 && todos.every((t) => t.done));

export function add(text: string) {
  if (!text.trim()) return;
  todos.push({ id: Date.now(), text: text.trim(), done: false });
}

export function toggle(id: number) {
  const todo = todos.find((t) => t.id === id);
  if (todo) todo.done = !todo.done;
}

export function remove(id: number) {
  const idx = todos.findIndex((t) => t.id === id);
  if (idx >= 0) todos.splice(idx, 1);
}

export function edit(id: number, text: string) {
  const todo = todos.find((t) => t.id === id);
  if (todo) {
    todo.text = text.trim();
    editingId = null;
  }
}

export function toggleAll() {
  const target = !allDone;
  todos.forEach((t) => (t.done = target));
}

export function clearDone() {
  todos = todos.filter((t) => !t.done);
}
```

## 15. The Reactive Graph

The Lattice compiler produces a **reactive graph file** — a generated artifact that captures the complete reactive topology of the project. Every signal, computed, effect, and template subscription is recorded with its dependencies and consumers. This file is deterministic (same source always produces the same graph), version-controlled, and used by the toolchain for cross-module resolution, cold-start elimination, impact analysis, and developer tooling.

The reactive graph is to Lattice's reactivity what `.d.ts` files are to TypeScript's types: the **extracted contract** for the reactive architecture of the code. TypeScript declarations tell consumers "here are the types this module exposes." The reactive graph tells consumers "here are the reactive bindings this module exposes, what they depend on, and what subscribes to them."

### 15.1 What's In It

The graph file is JSON, generated automatically by the Vite plugin after each successful build. It contains:

**Reactive bindings** — every signal and computed in the project, with its kind, scope (module-level or component-local), type, dependencies, and consumers:

```jsonc
{
  "version": 1,
  "signals": {
    "src/stores/counter.ts::count": {
      "kind": "signal",
      "type": "number",
      "scope": "module",
      "consumers": [
        "src/stores/counter.ts::doubled",
        "src/components/Counter.ts::template#0",
        "src/components/Dashboard.ts::template#2",
      ],
    },
    "src/stores/counter.ts::doubled": {
      "kind": "computed",
      "type": "number",
      "scope": "module",
      "dependencies": ["src/stores/counter.ts::count"],
      "consumers": ["src/components/Dashboard.ts::template#1"],
    },
    "src/stores/todos.ts::todos": {
      "kind": "signal",
      "type": "Todo[]",
      "scope": "module",
      "paths_accessed": ["*.done", "*.text", "*.id", "[collection]"],
      "consumers": [
        "src/stores/todos.ts::visible",
        "src/stores/todos.ts::remaining",
        "src/components/TodoApp.ts::template#0",
        "src/components/TodoApp.ts::effect#0",
      ],
    },
  },
}
```

**Effects** — every side effect, with its dependencies, cleanup status, and source location:

```jsonc
{
  "effects": {
    "src/components/Search.ts::effect#0": {
      "dependencies": ["src/stores/search.ts::query"],
      "has_cleanup": true,
      "location": { "line": 14, "col": 2 },
    },
  },
}
```

**Templates** — every template and its signal subscriptions, event handlers, and child components:

```jsonc
{
  "templates": {
    "src/components/Counter.ts::template#0": {
      "subscriptions": [
        { "signal": "src/stores/counter.ts::count", "binding": "text", "element": "button" },
      ],
      "event_handlers": 1,
      "child_components": [],
    },
  },
}
```

**Module summaries** — per-module export metadata for quick lookups (this is the data that drives cross-module signal resolution):

```jsonc
{
  "modules": {
    "src/stores/counter.ts": {
      "exports": { "count": "signal", "doubled": "computed", "increment": "function" },
      "imports": {},
      "signals": 1,
      "computeds": 1,
      "effects": 0,
    },
    "src/components/Counter.ts": {
      "exports": { "Counter": "component" },
      "imports": { "count": "src/stores/counter.ts", "increment": "src/stores/counter.ts" },
      "signals": 0,
      "computeds": 0,
      "effects": 0,
      "templates": 1,
    },
  },
}
```

### 15.2 Two Forms

The reactive graph manifests in two forms depending on context:

**Project-level graph** — the complete reactive topology of the entire application. Generated on build, lives at the project root, version-controlled. Contains every signal, computed, effect, template subscription, and their relationships across all files in the project. This is what enables tooling, CI diffing, cold starts, and dev tools visualization.

**Package-level graph** — the public reactive contract for a published library. A subset of the project-level graph, filtered to only exported signals and computeds with their kinds and types. Ships with the npm package alongside `.d.ts` files. This is what enables cross-library signal resolution (Section 9.4, tier 2 lookup).

```
@acme/ui-state/
├── dist/
│   ├── theme.js           # Compiled JavaScript (no embedded metadata)
│   ├── theme.d.ts          # TypeScript type contract
│   └── lattice.graph.json  # Reactive signal contract
└── package.json
```

Same format. The package-level version is just the project-level version filtered to public exports. The consuming compiler reads the package's graph file to discover signal exports — this is cleaner than embedding `__lattice_meta` in the JavaScript output because the reactive contract is fully separated from the runtime code.

### 15.3 What It Enables

**Instant cold starts.** When the dev server starts, the Vite plugin loads the graph file and pre-populates the in-memory signal metadata cache. Every file transformation has full cross-module signal knowledge from the very first request. No cold-start penalty, no "transform the dependency first" ordering.

**Breaking change detection.** When a signal export is renamed or removed, the compiler compares against the graph and reports which files are affected before the build runs:

```
Warning: Signal 'count' was removed from stores/counter.ts.
  Imported by: components/Counter.ts, components/Dashboard.ts
  Did you mean to rename it?
```

This works across library boundaries. Upgrading a Lattice dependency diffs the old and new graph files and reports which signal exports changed — a reactive migration report for free.

**Reactive architecture review in CI.** The graph file is version-controlled. Every PR that changes a signal binding, adds an effect, or changes a template's dependencies shows up as a graph file diff:

```diff
  "src/stores/auth.ts::user": {
    "kind": "signal",
    "scope": "module",
    "consumers": [
      "src/components/Header.ts::template#0",
      "src/components/Profile.ts::template#0",
+     "src/components/Sidebar.ts::effect#0",
+     "src/components/AdminPanel.ts::template#0"
    ]
  }
+ "src/stores/auth.ts::isAdmin": {
+   "kind": "computed",
+   "dependencies": ["src/stores/auth.ts::user"],
+   "consumers": [
+     "src/components/AdminPanel.ts::template#0"
+   ]
+ }
```

Reviewers see the architectural impact of changes at a glance: "This PR added an `isAdmin` computed that depends on `user`, and two new components now consume auth state."

**Impact analysis.** Before refactoring, query the graph to see the blast radius:

```bash
lattice graph consumers src/stores/auth.ts::user
# → src/stores/auth.ts::isAdmin (computed)
# → src/components/Header.ts::template#0 (text binding)
# → src/components/Profile.ts::template#0 (attribute binding)
# → src/components/Sidebar.ts::effect#0 (data fetch)
```

**Reactive linting.** Write static analysis rules against the graph:

- "No signal should have more than N direct consumers" (architectural bottleneck detection)
- "Effects should not depend on more than N signals" (complexity warning)
- "Every signal should have at least one consumer" (dead reactive state)
- "Template subscriptions should not read from more than N stores" (component doing too much)
- "No circular computed chains across modules"

These rules run against the graph file in CI — no application execution needed.

**Dev tools without running the app.** The browser extension or a standalone visualizer loads the graph file and renders the complete signal dependency graph as an interactive diagram. Navigate signals, see every consumer. Click an effect, see every dependency. No running application required.

**Stale dependency warnings.** When an effect's dependencies change between builds (it reads a signal it didn't before, or stopped reading one it used to), the compiler flags the change:

```
Info: Effect in components/Search.ts now depends on [query, isSearching].
  Previously depended on [query].
  New dependency: isSearching (added at line 14)
```

### 15.4 How It Updates

The graph file updates automatically as part of compilation. The developer never edits it manually.

**During production builds:** The Vite plugin aggregates per-file metadata from the Rust compiler after all files are transformed and writes the complete graph file. This is deterministic — same source code always produces the same graph.

**During dev server operation:** The plugin updates the graph incrementally as files are transformed, or writes it on server shutdown. The in-memory cache (used for cross-module resolution) is always current; the on-disk file may lag slightly during active development.

**If deleted:** The next build regenerates it from scratch. The graph file is a derived artifact, not a source of truth. Deleting it has no effect other than losing the cold-start optimization until the next build.

### 15.5 Where It Fits Architecturally

The graph file is generated by the Vite plugin (JavaScript side), not the Rust compiler. The Rust compiler transforms individual files and returns per-file metadata (`signalExports`, dependency information). The Vite plugin aggregates that metadata across all files and writes the graph.

```
Rust compiler:  file → { code, sourceMap, signalExports, dependencies }  (per-file, stateless)
Vite plugin:    aggregate all file metadata → graph file                  (project-level, stateful)
```

This keeps the Rust compiler stateless (pure function, string in, string out) while the Vite plugin manages project-level state. The graph file is the persisted form of that project-level state.

### 15.6 Relationship to `__lattice_meta`

The reactive graph subsumes the role of `__lattice_meta` for cross-module signal resolution. For published libraries, the graph file (filtered to public exports) ships as the reactive contract. The consuming compiler reads it directly instead of parsing `__lattice_meta` from compiled JavaScript.

`__lattice_meta` remains as a backward-compatibility fallback for cases where a library doesn't ship a graph file. New libraries should ship the graph file. Over time, `__lattice_meta` may be deprecated.

## 16. Fullstack Architecture (Future)

> **Note:** This section describes the architectural direction enabled by Lattice's platform-agnostic signal engine. It is not v1 scope. The client-side SPA is the v1 product. These capabilities are designed to be additive — nothing in v1 gets thrown away.

Lattice's signal engine has zero browser API dependencies. The same reactive primitives (`signal`, `computed`, `effect`, `watch`) work identically on the server.

### 16.1 Planned Capabilities

| Capability          | Depends on                                             | Status    |
| ------------------- | ------------------------------------------------------ | --------- |
| **Client-side SPA** | Signal engine + compiler + runtime                     | v1 target |
| **SSR**             | + `renderToString` mode, signal serialization          | Future    |
| **Hydration**       | + DOM attachment from serialized state                 | Future    |
| **Resumability**    | + Lazy hydration boundaries, partial signal recreation | Future    |
| **Islands**         | + Static analysis for interactive vs static components | Future    |
| **Meta-framework**  | + File-based routing, API routes, deployment adapters  | Future    |

### 16.2 Why the Architecture Supports This

Because signals are standalone reactive nodes (not tied to component instances like React's `useState`), hydration can recreate individual signals without re-running component functions. The dependency graph reconstructs itself from the source code — computed functions don't need serialization. This makes SSR → hydration → resumability a natural progression, not a rewrite.

**Critical architectural constraints:**

- `@lattice/signals` MUST have zero platform-specific dependencies. No `document`, no `window`, no Node-specific APIs. This is what enables the fullstack story. If this constraint is violated, the entire architecture falls apart.
- `crates/compiler` MUST have zero Node.js dependencies. It is a pure Rust library. The NAPI bridge in `napi/compiler` is the only place where Rust meets Node.js.
- The Rust compiler and the TypeScript signal engine are independent. The compiler transforms code that uses `@lattice/signals`, but does not depend on it at runtime. This separation means the signal engine can evolve without recompiling the compiler and vice versa.

### 18.1 Package API Boundaries

`@lattice/signals` is the raw reactive engine, written in TypeScript. It exports `signal`, `computed`, `effect`, `batch`, `untracked`, `toRaw`, `createOwner`, `runWithOwner`, and the public types. The `effect` primitive here is the auto-tracked side effect with immediate execution. The deep reactive proxy is part of this package — it wraps signal values transparently.

`@lattice/compiler` is the build-time code transformer, written in Rust with a thin JavaScript wrapper. It exports `transformSync` (and `transform` async) for use by the Vite plugin. End users never interact with this package directly.

`@lattice/runtime` (or the `lattice` re-export package) builds on top of `@lattice/signals` to provide `watch` (explicit source, old/new values, no initial run) and any other higher-level primitives. `watch` is a genuinely different primitive from `effect` — it has different semantics and is implemented separately, but built on the same signal engine internals.

The developer imports from `lattice` and gets everything. They never need to know which sub-package a primitive comes from.

- **Package setup:** package.json, build config, TypeScript declarations
- **Vanilla JS demo:** TodoMVC using raw signal/computed/effect with manual DOM — proves engine works end-to-end

**Done when:** All tests pass, public API is minimal and intentional, performance baselines established, package builds and the vanilla demo works.

### Phase 2: Deep Reactive Proxy (`@lattice/signals`)

Path-tracking Proxy wrapping backed by Phase 1 signals. The proxy maintains a map of path-based signals, created lazily on access. Reading a path subscribes the reader. Writing a path notifies only that path's subscribers (and parent paths).

**Key implementation details:**

- Single reactive store per signal, path-based signal tracking internally
- Recursive Proxy wrapping on nested object access, with views routing back to centralized tracking
- Automatic granularity selection: fine-grained for individual property access, collection-level for iteration
- Array method interception: mutating methods (push, splice, sort, etc.) trigger appropriate path notifications
- Structural mutations (splice, shift, unshift, sort, reverse) invalidate all index-based path subscribers
- Non-mutating array methods (filter, map, slice) return plain non-proxied arrays
- Identity stability via WeakMap cache (same nested object returns same view proxy)
- `toRaw()` strips all proxy wrapping recursively
- `untracked(fn)` suspends the tracking stack for the duration of the call

**Implementation approach:** Start with flat objects (property-level tracking), then layer in nested objects, then arrays with method interception. Collection-level tracking optimization last. Correctness first, optimization second.

**Acceptance tests (write before implementation):**

1. Dashboard with independent subtrees — mutation in one subtree does NOT notify effects in another
2. Form with field-level effects — changing one field does NOT trigger validation effects for other fields
3. List with item-level mutations — toggling one item does NOT re-evaluate other items' template computeds
4. Pass-a-reference-to-a-function — mutations through a passed proxy reference are reactive
5. Array structural mutations — splice correctly invalidates index-based subscribers
6. Array method returns — filter/map return plain arrays, mutations on returned arrays are NOT reactive
7. Identity stability — same nested path returns same proxy reference
8. toRaw — returns plain object, mutations on raw object are NOT reactive

**Done when:** All acceptance tests pass. No double-proxying. No memory leaks. Performance competitive with Vue 3's reactive proxy on equivalent benchmarks.

### Phase 3: Compiler — Signal Transform (`crates/compiler` + `napi/compiler`)

Set up the Rust compiler crate with OXC dependencies. Implement the signal transform as a `Traverse` impl. Parse TypeScript via `oxc_parser`, run scope analysis via `oxc_semantic`, identify `signal()` and `computed()` initializations, rewrite all reads and writes of those bindings to use `.value`. Implement edge case rules from Section 2.4. Emit `__lattice_meta` for signal/computed exports (backward compatibility). Return per-file signal metadata from `transformSync` for reactive graph aggregation. Code generation via `oxc_codegen` with source maps. Build NAPI bridge to expose `transformSync` to Node.js.

**Cross-module signal resolution:** The Vite plugin (TypeScript) maintains the in-memory metadata cache and generates the reactive graph file (Section 15). It passes imported signal metadata into each `transformSync` call. The Rust compiler uses this metadata to recognize cross-module signal bindings. On dev server startup, the Vite plugin loads the existing graph file to pre-populate the cache. On build completion, it writes the updated graph file.

**Done when:** `let count = signal(0); count++` compiles to correct signal operations. Object shorthand captures values. Destructuring is not transformed. Factory returns require explicit accessors. Cross-module signal imports are recognized from both local files and npm dependencies. Source maps are accurate. The Vite plugin runs and transforms files during dev and build. The reactive graph file is generated on build.

### Phase 4: Compiler — Template Pass-Through (`crates/compiler`)

Implement the custom HTML-in-tagged-template parser in Rust. Find `html` tagged template expressions in the OXC AST, extract template strings and expression holes, parse HTML structure, produce a template AST. Generate runtime DOM creation code (pass-through — not yet optimized). Wire into the Vite plugin.

**Done when:** A simple component renders in the browser. Template output matches expected DOM. Vite dev server runs with HMR.

### Phase 5: Compiler — Template Optimization (`crates/compiler`)

Generate direct signal-to-DOM subscriptions instead of effects. Static analysis to separate static vs dynamic parts. Keyed list reconciliation. Conditional branch optimization. CSS extraction and scoping.

**Done when:** Only affected DOM nodes update. Template purity violations produce compiler errors. Keyed lists handle add/remove/reorder efficiently. CSS is extracted and scoped at build time. Performance is competitive with Solid on js-framework-benchmark.

### Phase 6: Runtime (`@lattice/runtime`)

Component mounting, lifecycle hooks (`onMount`, `onDestroy`), ownership tree wired to DOM tree, `watch()` implementation, `mount()` API, component props.

**Done when:** Full TodoMVC app runs. Components mount, update, and unmount correctly. Ownership disposal cleans up all subscriptions. HMR works.

### Phase 7: Dev Tools & CLI

Browser extension. CLI scaffolding (`lattice create`). Build validation. Effect dependency visualization. Path tracking visualization for the deep proxy. Runtime dev-mode introspection (`__DEV__` parallel Set for live dependency enumeration, `__DEV__inspectDependents()` — deferred from Phase 1.5 because the reactive graph file provides static dependency information at build time; runtime introspection is only needed for live dev tools showing dynamic/conditional dependencies and current values).

**Done when:** Developers can inspect live signal values, see the dependency graph, see which paths are tracked by the deep proxy, see `effect()` dependencies, and diagnose update propagation issues.

**Done when:** A React developer can build a working app from the docs without external help.

## 20. Open Design Questions

These are decisions that don't need to be made yet but should be tracked:

1. **Auto-batching in effects:** Should `effect()` callbacks automatically batch their internal signal writes? Currently effects don't auto-batch — writing to two signals inside an effect triggers two propagations. Auto-batching would make it one. Needs a decision and explicit test coverage either way.

2. **Template language server:** Tagged templates have worse IDE support than JSX (syntax highlighting, autocomplete, error squiggles). A language server plugin at least as good as lit-html's is critical for adoption. This is the single biggest adoption risk.

3. **Compiler auto-accessor for factory returns:** Section 2.4 specifies explicit getter/setter for factory returns in v1. Automatic transformation (detecting signal bindings in return objects and generating getter/setter) is desirable but complex. This pattern comes up constantly in real applications — every composable, every store factory, every hook-style abstraction hits this. Should be prioritized as an early post-v1 enhancement.

4. **`shallow` option implementation timing:** The `{ shallow: true }` API is designed in (Section 5.9) but may not be implemented in Phase 2 if the path-tracking strategy performs well enough. Benchmark first, then decide.

5. **`Map` and `Set` support:** Deep proxy needs to decide whether `Map` and `Set` are proxied (with method interception) or left as-is with a warning. Deferring to Phase 2 implementation.

6. **Error boundaries:** The spec defines error handling for `computed` (Section 3.2) but has no story for component-level or template-level errors. What happens when a component's render function throws? When an effect throws? Options include: a dedicated `<ErrorBoundary>` component (React/Solid pattern), a `try` wrapper in the `html` template syntax, or a global error handler registered at mount time. This needs design before v1 — unhandled errors in components currently crash the entire app with no recovery path.

7. **Watch initial value semantics:** `watch` does not run on initialization (Section 4.6). The initial `oldValue` on first fire should be defined — likely `undefined`. An `immediate: true` option (matching Vue's API) may be needed for cases where developers want watch-like semantics with an initial run.

8. **Path-based signal garbage collection:** Lazily-created path signals in the deep proxy (Section 5.4) need a cleanup strategy for paths that are no longer observed. Options include: reference counting (dispose when observer count hits zero), periodic sweep, or WeakRef-based collection. This affects long-running SPAs with dynamic state shapes.

9. **Proxy compatibility with third-party libraries:** Some libraries use `Object.keys()`, `JSON.stringify()`, `instanceof`, or `typeof` checks in ways that may behave unexpectedly with Proxies. The `toRaw()` escape hatch handles explicit cases, but implicit cases (passing a proxied object to a library that internally inspects it) need testing and documentation. Consider which Proxy traps (`has`, `ownKeys`, `getPrototypeOf`) need special handling.

10. **Framework naming:** The framework name is subject to change. The current name "Lattice" originated from an earlier architectural direction. All internal references (metadata exports, scoping attributes, package names) should use a name that reflects the framework's current identity. This does not block implementation — it's a find-and-replace operation.

11. **Reactive graph file naming and location:** The graph file needs a name and extension. Options include `lattice.graph.json`, `{name}.reactive`, `.lattice/graph.json` (hidden directory with room for other artifacts), or a custom extension. The name should reflect what it is (the reactive topology) and be recognizable. Depends on framework naming decision (#10). For published packages, the file name/location needs to be discoverable by consuming compilers — either a `package.json` field or a convention.

12. **Reactive graph granularity:** The current design captures component-local signals in the project-level graph. An alternative is to only capture module-scoped (shared) signals in the project graph, since component-local signals are implementation details that don't affect other files. This would make the graph smaller and the CI diffs more focused on architectural changes. The package-level graph already only includes exports.

## 21. Design Decisions Log

Decisions made during the design process, preserved for context.

### 21.1 Deep Reactive Proxy vs Thin Proxy vs Compiler-Only Immutable Updates

**Decision:** Deep reactive proxy with path-level tracking.

**Alternatives considered:**

- **Thin proxy (mutation interception, no path tracking):** Any mutation anywhere notifies the entire signal. Consumers rely on computed equality checks to prevent unnecessary DOM updates. Simpler implementation (~200 lines vs ~2000), but requires developers to add `computed` intermediaries defensively to prevent over-notification in effects. The cognitive overhead of "am I reading too broadly?" mirrors React's `useMemo` problem — exactly what Lattice is designed to eliminate.

- **Compiler-only immutable updates (no proxy):** The compiler transforms `state.user.name = 'Bob'` into an immutable spread update. Elegant for static patterns but fundamentally breaks on dynamic property access, loop variables, callback parameters, and objects passed to helper functions. Reactivity becomes a property of how code is written rather than the data itself — refactoring a mutation into a helper function silently breaks reactivity. Not viable.

- **Separate `store()` primitive (Solid's model):** A second primitive for objects/arrays with deep reactivity. Adds a concept and violates the "one way to do things" principle. The deep proxy on `signal()` provides the same capability without the extra API surface.

**Rationale:** The deep proxy makes reactivity a property of the data, not the code. `state.user.name = 'x'` is reactive regardless of where the code lives — in a component, a helper function, a callback, a child component. No wrappers, no intermediaries, no thinking about subscription granularity. This is Lattice's core DX differentiator.

### 21.2 Cross-Module Signal Resolution via Metadata Export

**Decision:** Compile-time `__lattice_meta` export with two-tier lookup (in-memory cache + read from compiled output).

**Alternatives considered:**

- **In-memory only (no emitted metadata):** Works for local files but prevents publishing and consuming Lattice libraries across npm packages.
- **TypeScript type branding:** Requires the TypeScript compiler API during transforms, which is too slow for dev server HMR.
- **Runtime signal identity checking:** Adds runtime overhead and introduces two code paths (checked vs unchecked).
- **Convention-based (import source heuristic):** Can't distinguish signal exports from plain constant exports without actual knowledge.

**Rationale:** The `__lattice_meta` export is tiny, tree-shakeable (stripped by the compiler in final output), and enables library publishing without runtime cost. The in-memory cache handles the common case (local files) with zero disk I/O. Single-pass compilation in both dev and production because module graphs are resolved depth-first.

### 21.3 Rust + OXC Compiler Architecture

**Decision:** Write the compiler in Rust using OXC crates (`oxc_parser`, `oxc_traverse`, `oxc_codegen`). Expose to Node.js via NAPI bindings. The Vite plugin calls the native compiler via `transformSync`.

**Alternatives considered:**

- **Babel (JavaScript):** Most mature plugin ecosystem and easiest to write. But slow — 50-200ms per file for parse-transform-emit. Adequate for small projects, painful for large codebases. Many frameworks started on Babel and migrated to faster alternatives later. Development velocity advantage doesn't justify the permanent performance ceiling.

- **SWC from JavaScript (Rust parse/print, JS transform):** SWC's parser/printer at native speed, transforms in JS. ~2-3x faster than Babel end-to-end. SWC's AST format is less documented than Babel's, and the JS transform step loses the native speed advantage. A reasonable middle ground but still leaves performance on the table.

- **SWC Rust plugins (pure Rust):** Full native speed. But SWC's plugin API is less mature, documentation is thinner, and the plugin API has changed between versions. SWC is a viable alternative but OXC's architecture is better aligned with Lattice's needs (see below).

- **TypeScript Compiler API:** TypeScript's own transformer system. Verbose, poorly documented, no built-in visitor pattern. Performance between Babel and SWC. Type information available but slow to access (`ts.createProgram`). The type checking capability is potentially useful for template type safety but not worth the performance cost as the primary compiler.

- **Custom parser:** Building a TypeScript parser from scratch. Unnecessary — Lattice doesn't extend TypeScript syntax. Would take months to build and maintain as TypeScript evolves.

- **OXC from JavaScript (Rust parse/print, JS transform):** Similar to SWC-from-JS but with OXC's faster parser. Still loses native speed on the transform step. If Rust were not feasible, this would be the fallback.

**Rationale:** OXC provides the fastest available TypeScript parser (3x faster than SWC), a well-designed Traverse API for writing custom transforms, production-ready codegen with source maps, and distinct AST types (`BindingIdentifier` vs `IdentifierReference`) that map directly to the compiler's needs. OXC is the foundation of Rolldown (Vite's future bundler), making the Lattice compiler native to its build tool ecosystem.

Writing the compiler in Rust has a higher upfront cost than JavaScript/TypeScript. The `Traverse` trait pattern requires learning Rust idioms and working with OXC's arena allocator. But compiler code is written once and executed millions of times — the performance investment pays dividends on every file save for every developer using the framework. Sub-10ms file transforms make HMR feel instant and production builds of large projects complete in seconds rather than minutes.

The NAPI bridge keeps the integration clean: the Rust compiler is a pure library that takes a string and returns a string + metadata. The Vite plugin manages cross-module state in JavaScript. Neither side knows about the other's implementation details.

### 21.4 Reactive Graph as a Persistent Artifact

**Decision:** The compiler produces a reactive graph file — a generated, version-controlled JSON artifact that captures the complete reactive topology of the project (signals, computeds, effects, template subscriptions, and their dependency/consumer relationships).

**What it replaces:** The reactive graph subsumes `__lattice_meta` as the primary mechanism for cross-module signal resolution in published libraries. `__lattice_meta` remains as a backward-compatibility fallback.

**Alternatives considered:**

- **In-memory only (no persisted graph):** The original approach. Cross-module metadata exists only during a build and evaporates when the process exits. This works for compilation but provides no cold-start optimization, no CI diffing, no impact analysis, no offline dev tools visualization, and no cross-library upgrade safety checking. Every capability the graph enables would be impossible.

- **Embedded metadata only (`__lattice_meta`):** Metadata lives inside compiled JavaScript output. Works for cross-module resolution but mixes build-time information with runtime code. Requires stripping during bundle output. Doesn't provide project-wide topology — only per-module exports. Can't power CI diffing, impact analysis, or dev tools visualization.

- **Sidecar `.reactive` files per module:** Each compiled module gets a companion file with its signal metadata (analogous to `.d.ts` per `.js`). More granular than a single project-level graph but loses the cross-module relationship data — you'd know what each module exports but not who consumes it or what the effect dependency chains are. Also produces many small files instead of one structured artifact.

**Rationale:** The reactive graph is the natural consequence of having a compiler that sees the full reactive architecture. No other framework produces this information because no other framework has a compiler that tracks signal dependencies across modules, through templates, and into effects. Lattice's compiler already generates all of this data during transformation — the graph file is simply persisting it instead of discarding it.

The two-form design (project-level for tooling, package-level for library distribution) means one concept serves both the developer experience and the cross-package interop story. The graph file is analogous to `.d.ts` files for TypeScript: generated, version-controlled, and the contract that enables tools and consumers to understand the code without re-analyzing source.

No other framework has a persistent, version-controlled reactive dependency graph. This is a genuinely novel capability that enables tooling (CI reactive diffing, impact analysis, dead signal detection, reactive linting, offline dev tools) that doesn't exist anywhere in the frontend ecosystem.
